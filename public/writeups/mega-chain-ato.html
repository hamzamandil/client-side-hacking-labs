The Grand Chain: CSPT + PostMessage + OAuth = Account Takeover

Inspired by xssdoctor & matanber's combined research · The ultimate client-side bug chain

The Challenge

Three individually low-to-medium severity client-side bugs. None are reportable alone. But chained together, they produce a one-click, full account takeover worth $10K+ in bounty. This writeup walks through the complete chain — from discovery to exploitation.

The Three Bugs

Bug 1: Client-Side Path Traversal (Low-Medium)

The settings page loads configuration from /api/settings/{section} where section comes from a URL parameter. Path traversal lets you point the fetch at any endpoint:

// Settings page JavaScript
const section = new URLSearchParams(location.search).get('section') || 'general';
fetch('/api/settings/' + section)
  .then(r => r.json())
  .then(config => renderSettings(config));

// CSPT: ?section=../../api/files/attacker-upload
// Fetches attacker-controlled JSON instead of real settings

Alone: You can make the page fetch a different endpoint. So what? There's no sensitive data to steal, and the response just populates a settings form. Not reportable.

Bug 2: PostMessage Handler Without Origin Check (Low)

The OAuth callback error page has an analytics script that responds to postMessage requests with the current page URL:

// OAuth error page analytics
window.addEventListener('message', function(e) {
  // No origin check!
  if (e.data.type === 'getAnalytics') {
    e.source.postMessage({
      type: 'analyticsData',
      url: window.location.href  // Includes URL fragment!
    }, '*');
  }
});

Alone: It leaks the page URL to any postMessage sender. But the error page URL is just /oauth/callback?error=.... No sensitive data. Not reportable.

Bug 3: OAuth Supports Implicit Flow (Low)

The OAuth provider accepts response_type=token, returning the access token in the URL fragment:

# OAuth implicit flow accepted
GET /oauth/authorize?
  client_id=app&
  response_type=token&
  redirect_uri=https://app.target.com/oauth/callback

# After auth, redirects to:
# /oauth/callback#access_token=eyJ...&token_type=bearer

Alone: Implicit flow is deprecated but widely supported. The redirect_uri is validated, so you can't redirect to an external domain. Not reportable.

The Chain: Three Weak Bugs = Critical ATO

CSPT lets the attacker inject a response that triggers a postMessage to the parent window

PostMessage gadget on the OAuth error page leaks the URL (including the token fragment) to any origin

Implicit flow puts the access token in the URL fragment

Chain them: CSPT triggers OAuth flow → implicit flow puts token in URL → postMessage leaks the URL to attacker = token stolen

↓
5

↓

PostMessage gadget fires: The attacker's parent page sends {type:'getAnalytics'} to the error page iframe. The analytics handler responds with the full URL including #access_token=eyJ...

↓
7

Token stolen: Attacker extracts the access token from the postMessage response

↓
8

Account takeover: Attacker uses the token to change victim's email and password via the API

Why This Chain is Worth $10K+

Each bug alone is informational or low severity. But the chain achieves:

• One-click attack — victim just visits a URL

• No user interaction — everything happens automatically in iframes

• Full account takeover — email changed, password reset, attacker has full access

• Bypasses all protections — CSRF tokens, SameSite cookies, redirect_uri validation

This is the essence of client-side bug bounty: finding the chain that turns noise into signal.

Lessons for Bug Bounty Hunters

  • Never discard a low bug — keep a list. Every CSPT, every postMessage leak, every open redirect is a potential chain link
  • Map the connections — which pages embed which iframes? Which scripts handle postMessage? Where does OAuth redirect?
  • Think in attack graphs — Bug A lets you control data, Bug B leaks URLs, Bug C has the token. Connect the dots.
  • Test the full chain — A chain that works in theory but fails in practice isn't a finding. Build the PoC, prove it works end-to-end.
  • Report the chain, not the bugs — The report should show the full ATO, with each bug explained as a step. Programs pay for impact, not for individual weaknesses.

Bug Bounty Tip

The researchers who consistently find $10K+ chains don't have magical tools. They have a mental inventory of every small bug on the target. When they find a new low-sev issue, they immediately check: "Does this connect to anything I already know?" The best way to build this skill is to practice chains in labs, then apply the same thinking to real targets.
