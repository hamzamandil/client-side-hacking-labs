Cache Deception + Client-Side Path Traversal = Zero-Interaction ATO

Based on zere.es' research · Two medium bugs chain into one-click account takeover

Summary

Neither Cache Deception nor CSPT was exploitable alone on the target. Cache Deception needed an auth header that browsers can't send automatically. CSPT allowed path traversal on an authenticated request but didn't land anywhere useful. Combined, they produced a zero-interaction account takeover: one click from the victim, one unauthenticated fetch from the attacker, full token theft.

Background: Cache Deception in One Paragraph

CDNs like Cloudflare, Akamai, and Fastly cache static assets to reduce origin load. They typically identify "static" content by file extension — .css .js .png .jpg .woff. If the origin server routes /page.css to a dynamic authenticated handler (which returns user-specific data) AND the CDN caches the response because of the extension, the next request to /page.css gets served the cached authenticated content — without the CDN ever checking who the requester is.

Omer Gil documented this pattern in 2017. Twelve years later, it's still a reliable bug class.

Bug 1: Cache Deception on the Token Endpoint

The researcher found /v1/token, which returned the user's JWT:

GET /v1/token HTTP/1.1
Host: api.example.com
X-Auth-Token: <redacted>

HTTP/1.1 200 OK
Cache-Control: no-cache, no-store, must-revalidate
X-Cache: Miss from cdn
{"accessToken": "eyJ...", "scope": "admin", ...}

Good: Cache-Control: no-store. But testing /v1/token.css revealed the CDN overrode it:

GET /v1/token.css HTTP/1.1
Host: api.example.com
X-Auth-Token: <redacted>

HTTP/1.1 200 OK
Cache-Control: max-age=86400, public  ← CDN overrode!
X-Cache: Hit from cdn
{"accessToken": "eyJ...", "scope": "admin", ...}

The CDN cached the sensitive response for 24 hours. Problem: the endpoint required the X-Auth-Token custom header, which browsers cannot send automatically. You can't just link a victim to /v1/token.css and expect their browser to auth. Stuck.

Bug 2: CSPT in the User Info Endpoint

A separate piece of client-side JavaScript built an authenticated URL from user input:

const userId = new URLSearchParams(location.search).get('userId');
const apiUrl = `https://api.example.com/v1/users/info/${userId}`;
fetch(apiUrl, {
  headers: { 'X-Auth-Token': authToken }  // Auto-set from app state
});

The userId flows into the URL path. Classic CSPT. The researcher tried traversing to interesting endpoints and got 404s — nothing reachable via traversal was sensitive. Stuck.

The Insight

CSPT gives us control over the path of an authenticated request.

Cache Deception needs that authenticated request to land at a cacheable URL.

Chain them: use CSPT to send the authenticated token request to /v1/token.css. The browser attaches X-Auth-Token (the app's own code does that). The origin returns the token. The CDN caches it. The attacker reads the cache.

Building the Chain

The Crafted URL

https://example.com/user?userId=../../../v1/token.css

What the Browser Does

  1. Victim clicks the link
  2. Client JS reads userId=../../../v1/token.css
  3. Builds: https://api.example.com/v1/users/info/../../../v1/token.css
  4. Browser normalizes the URL → https://api.example.com/v1/token.css
  5. Browser sends the fetch WITH X-Auth-Token (the app's own header)
  6. Origin sees /v1/token.css — routing matches the /v1/token* handler
  7. Origin returns the user's token
  8. CDN sees .css, caches response for 24h

What the Attacker Does

# No auth needed — fetch the cached response
curl https://api.example.com/v1/token.css

HTTP/1.1 200 OK
X-Cache: Hit from cdn
{"accessToken": "eyJ...", "scope": "admin", ...}

Token stolen. Attacker uses it to change the victim's email, reset password, authorize new OAuth grants, or exfiltrate data. Full account takeover.

Why This Pattern Is Dangerous

Two mitigations that each look sufficient in isolation fail when composed. The origin team hears "use Cache-Control: no-store" and does so — but the CDN overrides based on extension. The frontend team hears "don't use user input in fetch paths" — but their use case feels safe because there's nothing sensitive reachable by traversal. Security in depth requires treating these as a single system.

Variants You Should Test

1. Extension Variations

  • .css, .js, .json, .png, .jpg, .svg, .webp, .woff, .woff2, .pdf
  • .map (sourcemap — sometimes cached)
  • Different cases: .CSS, .Js — some CDNs are case-insensitive

2. Path Delimiter Quirks

  • /page/profile.css — static-looking path under dynamic parent
  • /page;x.css, /page%00.css, /page#.css — origin and CDN may parse differently
  • Wildcard traversal: /$/profile.css — the ChatGPT bug

3. Real-World CSPT Sinks

  • fetch(\`/api/users/${id}\`)
  • fetch(\`/api/${resource}/details\`)
  • XMLHttpRequest.open('GET', basePath + input)
  • GraphQL client building client.query({ variables: { path: userInput } }) when path is concatenated server-side

Bug Bounty Tip

When you find cache deception but no way to auto-send auth headers from a victim's browser, don't give up. Look for CSPT in the app's own JavaScript. The app sends its own auth headers. You control the path. Any authenticated endpoint the app reaches becomes a cache deception target.

Conversely: when you find CSPT but no sensitive endpoints reachable via traversal, don't give up. Look for cache deception. A non-sensitive endpoint that returns user data + a cacheable extension = exfil primitive.

Related: Cache Poisoning

Cache deception = victim fills the cache with their data, attacker reads it.

Cache poisoning = attacker fills the cache with malicious content, victim reads it.

Both exploit mismatched cache keys. Deception exploits missing auth validation on cache hit. Poisoning exploits unkeyed headers that influence response content. Together they cover the full spectrum of "cache trusts too much."

Common Unkeyed Headers for Poisoning

  • X-Forwarded-Host — host used for URL building
  • X-Forwarded-Scheme — scheme mismatch
  • X-Host, X-Original-URL, X-Rewrite-URL
  • Forwarded (RFC 7239)
  • Custom app headers leaking into response

Tools: Web-Cache-Vulnerability-Scanner, Param Miner.
