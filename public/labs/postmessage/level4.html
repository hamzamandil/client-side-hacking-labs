PostMessage #4: iframe Sandbox Escape Hard

Objective

Execute alert('XSS') in the context of the PARENT page (not the iframe). The parent page trusts messages from iframes with a specific origin and renders them. The iframe content is user-controlled via a URL parameter.

Hints

The parent page embeds an iframe that loads /labs/postmessage/widget-frame.html. The iframe sends data to the parent via postMessage, and the parent renders it. The parent checks that messages come from http://localhost:3000.
The iframe page accepts a ?content= parameter and renders it. But the iframe has sandbox="allow-scripts allow-same-origin". Since it's same-origin, its postMessages pass the parent's origin check. Control the iframe content by setting the content parameter.
1. The iframe loads with your content parameter
2. Your content includes a script that calls parent.postMessage()
3. The parent receives the message — it passes the origin check because the iframe is same-origin
4. The parent renders the message content via innerHTML
Try: ?content=<script>parent.postMessage(JSON.stringify({type:'widgetData',html:'<img src=x onerror=alert(\\\"XSS\\\")>'}),'*')</script>

Vulnerable Code (Parent)

// Parent trusts same-origin iframe messages
window.addEventListener('message', (e) => {
  if (e.origin !== location.origin) return;

  var data = JSON.parse(e.data);
  if (data.type === 'widgetData') {
    // Renders widget HTML from iframe
    widget.innerHTML = data.html;
  }
});

// Iframe loads user-controlled content param
<iframe sandbox="allow-scripts allow-same-origin"
  src="/widget-frame.html?content=USER_INPUT"
></iframe>

Real-world context: Many apps use iframes for isolating third-party content (embeds, widgets, ads). The sandbox attribute restricts iframe capabilities, but allow-scripts allow-same-origin together effectively negate the sandbox — the iframe can run scripts AND it's treated as same-origin, so its postMessages pass origin checks.

Bug bounty tip: Look for iframes with sandbox="allow-scripts allow-same-origin" where the iframe URL has user-controlled parameters. This combination is explicitly warned against in the HTML spec because it allows the iframe to remove its own sandbox.
