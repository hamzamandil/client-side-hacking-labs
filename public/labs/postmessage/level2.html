PostMessage #2: indexOf Origin Bypass Medium

Objective

Execute alert('XSS') by bypassing the origin check that uses indexOf(). The handler checks if the origin contains "localhost" but the check is weak.

Hints

origin.indexOf('localhost') returns true for ANY origin containing "localhost" — like http://localhost.evil.com or http://evil.com?localhost. The check doesn't verify the origin is EXACTLY from localhost.
Since we're testing locally, the origin IS localhost — but the real lesson is understanding why indexOf is flawed. For this lab, you can exploit it directly from the attacker page since both run on localhost. In a real attack, you'd register a domain like localhost.attacker.com.
After bypassing the origin check, notice the handler uses eval(data.callback) to call a function by name. If you control the callback parameter, you can inject arbitrary JavaScript. Send: {"action":"render","callback":"alert('XSS')"}

Vulnerable Code

window.addEventListener('message',
  function(e) {
    // Weak origin check with indexOf
    if (e.origin.indexOf('localhost') === -1) {
      return; // "blocked"
    }

    var data = JSON.parse(e.data);
    if (data.action === 'render') {
      // Calls a "callback" function
      eval(data.callback + '()');
    }
  }
);

Real-world context: Developers often try to validate origins with indexOf(), includes(), endsWith(), or regex that doesn't anchor properly. Common bypasses:

  • indexOf('example.com') → bypass with example.com.evil.com
  • endsWith('.example.com') → bypass with notexample.com
  • /example\.com/ → bypass with example.com.evil.com
  • Only event.origin === 'https://example.com' is truly safe
