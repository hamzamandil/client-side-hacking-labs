PostMessage #3: eval() Message Handler Medium

Objective

Execute alert('XSS') by exploiting an admin debug tool that accepts commands via postMessage and passes them to eval(). The origin check exists but can be bypassed via the wildcard pattern.

Hints

The handler checks if origin matches /^https?:\/\/(.+\.)?localhost/. This regex allows any subdomain of localhost. Since we're on localhost, our messages pass. But in a real scenario, evil-localhost.com would also match because the regex doesn't properly anchor the domain.
The handler expects messages with {"cmd": "command_here"}. The command goes through a "whitelist" check that only allows commands starting with console., document.title, or Math.. But the check only looks at the START of the string.
Send a command that starts with a whitelisted prefix but contains malicious code after: {"cmd":"console.log(1);alert('XSS')"} or {"cmd":"Math.max(1,2),alert('XSS')"}. The startsWith check passes, and eval runs the entire string including your payload.

Vulnerable Code

const ALLOWED = [
  'console.', 'document.title', 'Math.'
];

window.addEventListener('message', (e) => {
  // Regex origin check (bypassable)
  if (!/^https?:\/\/(.+\.)?localhost/.test(e.origin))
    return;

  let { cmd } = JSON.parse(e.data);

  // "Whitelist" check (only checks prefix!)
  if (!ALLOWED.some(a =>
    cmd.startsWith(a)
  )) return;

  // VULNERABLE: eval with "whitelisted" command
  eval(cmd);
});

Local Test (send from this page):

Real-world context: Debug/admin panels sometimes accept commands via postMessage for cross-frame communication. Whitelist-based validation using startsWith or indexOf is dangerous because it doesn't prevent command chaining with semicolons, commas, or logical operators.

Bug bounty tip: Always check for eval(), new Function(), or setTimeout(string) in postMessage handlers. Even with "whitelisting", JavaScript's flexible syntax makes prefix checks trivially bypassable.
