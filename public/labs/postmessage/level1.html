PostMessage #1: No Origin Check Easy

Objective

Execute alert('XSS') by sending a malicious postMessage to this page from the attacker page. The widget accepts messages without checking the origin.

Hints

A page can send messages to another window using targetWindow.postMessage(data, '*'). The receiving page listens with window.addEventListener('message', handler). If the handler doesn't check event.origin, ANY page can send it messages.
Open this lab page, then open the attacker page in another tab. The attacker page uses window.open() to get a reference to this page, then sends it a postMessage with HTML that gets rendered via innerHTML.
Click "Open Attacker Page" below. It will open in a new tab. From there you can send postMessages to this page. The message is expected to be JSON with a type and content field. The content is rendered via innerHTML. Send: {"type":"update","content":"<img src=x onerror=alert('XSS')>"}

Vulnerable Code

// Listens for widget update messages
window.addEventListener('message',
  function(event) {
    // BUG: No origin check!
    // Should be: if(event.origin !== expected) return;

    var data = JSON.parse(event.data);
    if (data.type === 'update') {
      document.getElementById('widget')
        .innerHTML = data.content;
    }
  }
);

Real-world context: Many websites use postMessage for cross-origin communication — chat widgets, analytics, payment iframes, SSO flows. If the message handler doesn't validate event.origin, any page that can get a reference to the window (via window.open, iframe, or window.opener) can send malicious messages.

Bug bounty tip: Search for addEventListener('message' or onmessage and check if the handler validates the origin. Then look at what it does with the data — innerHTML, eval, location changes, etc.
