Hint: CloudDocs postMessage Iframe XSS

The page builds iframes from postMessage data using string concatenation. No origin check, no sanitization. Break out of an attribute.
1
Find the listener›
View source. Find window.addEventListener('message', ...). The handler checks if event.data.message === 'e:openFloatingPage' but does NOT check origin.
2
Spot the sink›
The handler builds an HTML string with string concatenation: 
'<iframe src="' + d.url + '" name="' + d.name + '" id="fpf-' + d.id + '" ...>'
Then does container.insertAdjacentHTML("beforeend", panelHtml). Three injection points: url, name, id.
3
Attribute breakout›
To escape the src="..." attribute, inject a " followed by an event handler. The iframe needs to actually load for onload to fire, so use a URL that returns something (even 404 is fine).
4
X-Frame-Options›
Many URLs are framed — the server's /api/download endpoint has no X-Frame-Options. Use it as the iframe src. Or about:blank works too for breakout-only payloads.
5
Send the message›
Open this page, then in its own DevTools console run postMessage(...). Or build an attacker page that iframes this one and posts.

In this page's DevTools console:

postMessage({
  message: 'e:openFloatingPage',
  data: {
    id: 'pwn',
    name: 'pwn',
    url: '/api/download?p=x" onload="alert(\'XSS\')'
  }
}, '*');

Resulting HTML:

<iframe src="/api/download?p=x" onload="alert('XSS')"
   name="pwn" id="fpf-pwn" ...></iframe>

Iframe loads /api/download, onload fires, alert pops.
