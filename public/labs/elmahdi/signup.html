Hint: Cookie DOM XSS — /signup

This is the unpatched endpoint. Same JS code as /login but the server doesn't overwrite the cookie. Plant your payload in the lang cookie and reload.
1
Identify the injection context ›
View page source. The inline script contains: 
var page = { config: { lang: "{YOUR_COOKIE}", ... } };
You need to break out of the double-quoted string and inject a statement.
2
Why is this a sink? ›
The script element's innerHTML is assigned after the element is already in the DOM. Unlike setting .textContent or .src, innerHTML on a script tag replaces its contents, and browsers re-parse and execute the new code (in some cases) — or the script has already been evaluated and the JS string concatenation is the actual vector. Here, the interpolated cookie value becomes part of executable JS.
3
Cookie format ›
Cookies can't contain some characters directly (commas, semicolons, spaces need encoding or quoting). You can set the cookie via document.cookie in the console. Format: document.cookie = 'lang=PAYLOAD; path=/';
4
Breakout payload ›
To escape the JS string and inject code, you need to close the quote, run your JS, and comment out the rest: "; alert(1); //

In DevTools console on this page:

document.cookie = 'lang="; alert("XSS"); //; path=/';
location.reload();

Result: the inline script becomes var page = { config: { lang: ""; alert("XSS"); //", ... } }; and your alert fires.
