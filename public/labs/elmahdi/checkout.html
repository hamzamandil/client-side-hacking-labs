Hint: Chrome innerHTML Quirk

The page accepts postMessage events and "safely" decodes HTML-encoded JSON using a detached <p> element. The dev assumed detached elements can't execute scripts. Chrome says otherwise.
1
Find the message handler ›
View source or open DevTools. There's a window.addEventListener('message', ...) that processes string data as JSON. It has no origin check.
2
Understand the "safe" path ›
If the message contains &quot;, the handler creates a detached <p> element, sets p.innerHTML = message, then reads p.innerText. The element is never appended to the DOM. Seems safe, right?
3
The Chrome quirk ›
When you assign innerHTML that contains an <img> tag, Chrome immediately tries to load the src attribute — even if the element isn't attached to the DOM. If the src fails to load, onerror fires. 
const p = document.createElement('p');
p.innerHTML = '<img src=x onerror=alert(1)>';
// In Chrome, alert(1) fires even though p is never appended!
4
Trigger the decoder path ›
To reach the detached element code path, your payload needs to contain &quot;. So prefix your img tag with that string. The actual message is processed as a string, so any content works — JSON parsing can fail after the XSS fires.
5
Deliver the payload ›
Either: (a) open this page then run window.postMessage(...) in its own console, or (b) set up an attacker page that frames this one and posts the message. For testing, the console method is fastest.

In this page's DevTools console:

window.postMessage(
  '&quot;<img src=x onerror=alert("XSS_QUIRK")>',
  '*'
);

The handler sees &quot;, takes the "safe" decoder path, sets innerHTML on a detached <p>, and Chrome fires the img's onerror anyway.
