Hint: Cookie DOM XSS — /login

This page IS vulnerable in the JS code, but the server overwrites the lang cookie on every response. Your payload gets wiped before the code runs. The bug is the inconsistent defense — check the signup endpoint instead.
1
Find the sink ›
Open DevTools and look at the inline script. The lang cookie value is read and injected into a <script> element via innerHTML — which parses and executes the JS.
2
Why doesn't it work on /login? ›
Open Network tab. The response to /elmahdi/login includes a Set-Cookie: lang=en header that overwrites your malicious cookie before the JS reads it. The defense is server-side cookie rewriting.
3
Find the bypass ›
The developer patched /login but forgot to patch /signup. Same vulnerable JS, but no Set-Cookie overwrite. Visit /elmahdi/signup and try your cookie payload there.
4
Craft the payload ›
The value is interpolated into a JS string literal inside a <script> tag. To escape, close the string, inject code, reopen: "; alert(document.domain); //. Your cookie becomes: lang="; alert(1); //

1. In DevTools console on this page, set the cookie:

document.cookie = 'lang="; alert("XSS_SOLVED"); //; path=/';

2. Navigate to the vulnerable signup endpoint:

location.href = '/elmahdi/signup';

The signup page reads the lang cookie, interpolates it into the inline script, and your injected code executes.
