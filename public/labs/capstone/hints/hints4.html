SecureAuth - Hints

Only reveal what you need. Try to solve it yourself first.

Click "Sign in with SSO" and watch the URL. The app redirects to /oauth/authorize with several parameters: client_id, response_type, redirect_uri, and state. After authenticating, you're redirected back with a code. What if you could control where the code is sent?
The OAuth provider validates redirect_uri using a prefix check — it only verifies that the URI starts with a registered callback URL. This means you can append path traversal sequences to redirect the authorization response to a different page on the same origin.
The server has an open redirect at /redirect?url=.... You can chain this: use path traversal in the redirect_uri to reach the open redirect, which then sends the code to an attacker-controlled page. Also try changing response_type to token to get the access token directly in the URL fragment.
With response_type=token, the OAuth provider puts the access token in the URL fragment (after #). This fragment is accessible via JavaScript but not sent to the server. If you redirect the user to a page you control, your JavaScript can read the fragment and steal the token.
Craft this URL: /oauth/authorize?client_id=vuln-app&response_type=token&state=evil&redirect_uri=http://localhost:3000/oauth/callback/../../../labs/attacker/steal.html. After authentication, the token lands on the steal page in the URL fragment. The steal page reads it and can use it to call /oauth/userinfo.
