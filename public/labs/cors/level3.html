CORS #3: postMessage PII Leak via Framing Medium

Objective

A form widget broadcasts submission data via postMessage with targetOrigin="*" and has no X-Frame-Options header. Frame the widget page and intercept the PII (name, email, phone, SSN) when the form is submitted.

Hints

Two issues combine: (1) The widget at /api/form-widget uses postMessage(data, '*') — broadcasting form data to ANY parent window. (2) There's no X-Frame-Options or frame-ancestors CSP, so anyone can iframe the widget.
Embed the widget in an iframe on your attacker page. Listen for postMessage events. When the user submits the form, the widget broadcasts the PII to your parent page. You don't need XSS — just framing + listening.
window.addEventListener('message', (e) => { if (e.data.type === 'formSubmission') { steal(e.data.data); } }); — The parent page receives ALL postMessages from the iframe because targetOrigin is '*'.

Vulnerable Code

// Widget form submission handler
form.addEventListener('submit', (e) => {
  e.preventDefault();
  var data = new FormData(form);

  // VULNERABLE: broadcasts PII to ANY parent
  window.parent.postMessage(
    { type: 'formSubmission', data: formData },
    '*'  // Any origin!
  );
});

// MISSING: No X-Frame-Options header
// MISSING: No Content-Security-Policy frame-ancestors

Real-world context: This pattern was found in HubSpot, Typeform, and Tawk.to widgets. When a widget uses postMessage(data, '*') and the page is frameable, ANY website can embed the widget and steal form submissions. Impact: PII theft (names, emails, phone numbers, SSNs, credit cards) from any form that uses the vulnerable widget. Found on TryHackMe, SHEIN, BlueSnap, and others.

Bug bounty tip: Search for postMessage( with '*' in third-party widget scripts. Then check if the page has X-Frame-Options or frame-ancestors CSP. If both are missing, you have a frameable PII leak.
