CORS #1: Wildcard with Credentials Medium

Objective

Steal the victim's secret API data by exploiting a CORS misconfiguration. The API at /api/cors-demo/secrets reflects any Origin header with Access-Control-Allow-Credentials: true. Read the victim's data from an attacker page.

Hints

CORS (Cross-Origin Resource Sharing) controls which origins can read API responses. When a server reflects any Origin with Access-Control-Allow-Credentials: true, any website can make authenticated requests and READ the response — stealing user data.
From an attacker page, use fetch('/api/cors-demo/secrets', {credentials: 'include'}). Because the server reflects the attacker's origin with credentials allowed, the browser lets the attacker page read the response containing the victim's secrets.
1. Log in as victim to establish a session cookie
2. Click "Launch CORS Attack" — this simulates an attacker page making a credentialed fetch
3. The API returns the victim's secrets because it trusts any origin
4. The attacker page reads and displays the stolen data

Vulnerable Code

// Server: reflects ANY origin
app.get('/api/cors-demo/secrets', (req, res) => {
  const origin = req.headers.origin;
  res.header('Access-Control-Allow-Origin',
    origin);
  res.header('Access-Control-Allow-Credentials',
    'true');
  res.json({ apiKey, ssn, sessions });
});

// Attacker page:
fetch('/api/cors-demo/secrets', {
  credentials: 'include'
}).then(r => r.json())
  .then(data => exfiltrate(data));
