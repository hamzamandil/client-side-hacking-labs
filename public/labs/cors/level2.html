CORS #2: Subdomain Takeover + CORS Chain Expert

Objective

The API trusts any *.localhost subdomain origin. A subdomain (partners.localhost) has a dangling CNAME. Simulate claiming the subdomain, then use the trusted origin to steal credentialed API data. This is a P1 chain.

Hints

The API at /api/cors-demo/subdomain-data validates origin with a regex: /^https?:\/\/([a-z0-9-]+\.)*localhost(:\d+)?$/. This trusts ANY subdomain of localhost — like http://evil.localhost:3000.
In a real scenario, partners.company.com has a CNAME pointing to a third-party service (e.g., Unbounce, Heroku, GitHub Pages) that's no longer configured. An attacker can claim that service and host their own content on the subdomain. Since the API trusts *.company.com, the attacker's page can make credentialed requests.
Since we can't actually create subdomains locally, this lab simulates the attack. Click "Simulate Subdomain Takeover" to demonstrate the CORS chain. In a real engagement, you'd:
1. Find dangling CNAMEs via dig
2. Claim the service (create Heroku app, GitHub Pages, etc.)
3. Host your CORS exploit on the claimed subdomain
4. CORS trusts your origin → steal data
