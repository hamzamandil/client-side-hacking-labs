OAuth #2: Dirty Dancing - State + PostMessage Hard

Objective

Steal the victim's OAuth token using the "dirty dancing" technique. Break the OAuth flow by sending an invalid state, then use the postMessage analytics gadget on the error page to steal the token from the URL.

Hints

1. Start an OAuth flow with response_type=token (implicit flow) so the token goes in the URL fragment
2. Use an invalid/attacker-controlled state parameter
3. When the victim authenticates, they get redirected with the token in the URL
4. The callback page sees invalid state and shows an error, BUT the token is still in the URL
The OAuth error page loads a jQuery analytics script that has a postMessage listener. It responds to messages with {type:'getAnalytics'} by sending back the current page URL — which contains the token! Open the error page in an iframe or popup, then send it a postMessage to steal the URL.
1. Open /oauth/authorize?client_id=vuln-app&response_type=token&state=INVALID&redirect_uri=http://localhost:3000/oauth/callback in a popup
2. Victim authenticates (pre-filled for demo)
3. Gets redirected to /oauth/callback?error=invalid_state#access_token=TOKEN
4. Error page has analytics gadget listening for postMessage
5. From attacker page, send {type:'getAnalytics'} to the popup
6. Error page responds with URL containing the token
7. Parse the token from the URL!

Real-world context: The "Dirty Dancing" technique was discovered by Frans Rosen. It exploits the fact that OAuth tokens/codes remain in the URL even after errors. Combined with postMessage gadgets (analytics, chat widgets, error tracking), an attacker can extract these tokens from the error page without needing XSS. This works on major OAuth providers including Google, Microsoft, and Facebook.
