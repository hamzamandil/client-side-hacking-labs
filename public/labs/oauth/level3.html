Objective

Perform a complete dirty dancing attack combining THREE techniques: response_type switching, redirect_uri manipulation, and postMessage token theft. Then use the stolen token to change the victim's email (full ATO).

Hints

Use response_type=code,id_token (hybrid flow). This puts the code in the query string AND an id_token in the fragment. Different response types use different response modes — the token goes in the URL fragment which is harder to protect server-side but accessible via JavaScript.
The app has an open redirect at /redirect?url=.... Chain: use the legitimate callback URL with an open redirect parameter: redirect_uri=http://localhost:3000/oauth/callback?next=/redirect?url=/labs/attacker/steal.html. Wait — actually the redirect_uri validation is prefix-based, so use path traversal instead.
1. Use response_type=token to get token in fragment
2. Use redirect_uri path traversal to redirect to a page you control that has a postMessage listener
3. OR: use the simpler approach — same-origin popup + direct URL read
4. Extract the token from the URL
5. Call /api/user/change-email and /api/user/change-password using the victim's session
Full ATO!
Even simpler: the OAuth flow sets a session cookie when the victim authenticates. After the OAuth redirect, the victim is logged in via cookie. You can CSRF the email change because it has no CSRF token. So: 1) OAuth popup authenticates victim, 2) Use the session cookie to change email via fetch with credentials:include, 3) No token needed — just the cookie!
