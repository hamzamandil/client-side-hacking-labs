OAuth #1: Open Redirect Token Leak Medium

Objective

Steal the OAuth authorization code by manipulating the redirect_uri parameter. The OAuth provider has weak redirect_uri validation that only checks if the URI starts with the registered domain.

Make the authorization code get sent to an attacker-controlled endpoint.

Hints

The OAuth provider checks if redirect_uri starts with one of the registered URIs. But it doesn't check for path traversal or appended paths. The registered URI is http://localhost:3000/oauth/callback.
Try appending to the redirect URI: http://localhost:3000/oauth/callback/../../../labs/attacker/steal.html. The browser resolves this to http://localhost:3000/labs/attacker/steal.html but the server-side check passes because it starts with the registered URI prefix.
1. Craft an OAuth URL with manipulated redirect_uri:
/oauth/authorize?client_id=vuln-app&response_type=code&redirect_uri=http://localhost:3000/oauth/callback/../../../labs/attacker/steal.html&state=abc
2. When the victim authorizes, the code gets sent to your steal page
3. The steal page reads ?code= from the URL and displays it

Vulnerable Code

// OAuth provider: redirect_uri check
let redirectValid = client.redirect_uris
  .some(uri =>
    redirect_uri.startsWith(
      uri.split('?')[0]
    )
  );

// Only checks if URI STARTS with registered
// Allows: /callback/../attacker/page
// Allows: /callback/../../evil

Real-world context: OAuth redirect_uri validation is one of the most common sources of account takeover bugs. Common bypasses:

  • Path traversal: /callback/../attacker
  • Subdomain: callback.attacker.com if check is endsWith
  • Parameter: /callback?redirect=evil.com
  • Fragment: /callback#@evil.com
  • Open redirect chaining: /callback?next=//evil.com
  • Case variance: /CallBack vs /callback
