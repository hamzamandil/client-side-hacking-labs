DOM XSS #5: DOM Clobbering to XSS Hard

Objective

Execute alert('XSS') by using DOM clobbering to override a JavaScript variable. The app allows limited HTML in user bios but uses DOMPurify (simulated). However, the code references a global variable that can be clobbered.

Hints

HTML elements with id or name attributes create global variables on window. For example, <img id="foo"> makes window.foo reference that element. This can override existing JavaScript variables.
The code uses window.globalConfig.defaultAvatar which falls back to a safe URL. But if you can clobber globalConfig with an HTML element that has a defaultAvatar property... Anchors (<a>) have properties that return their attribute values as strings!
Use nested anchors with a form: <form id="globalConfig"><img name="defaultAvatar" src="x" onerror="alert('XSS')"></form> — Wait, that won't work with DOMPurify. Instead, clobber with <a id="globalConfig"><a id="globalConfig" name="defaultAvatar" href="javascript:alert('XSS')">. DOMPurify allows anchor tags and href! The code later uses this value as an image src which won't trigger JS, but check what else the code does with it...
The code also uses globalConfig.analyticsUrl in a script src. Clobber globalConfig to control the script URL. Use: <a id="globalConfig"><a id="globalConfig" name="analyticsUrl" href="data:text/javascript,alert('XSS')">. Two anchors with the same id create an HTMLCollection, and named access on the collection returns the element with that name.

Vulnerable Code

// Global config — if not already set
if (!window.globalConfig) {
  window.globalConfig = {
    analyticsUrl: '/js/analytics.js',
    defaultAvatar: '/img/default.png'
  };
}

// Later, after user HTML is inserted:
var avatar = window.globalConfig.defaultAvatar;
var script = document.createElement('script');
script.src = window.globalConfig.analyticsUrl;
document.body.appendChild(script);

Real-world context: DOM clobbering exploits how browsers create global variables from HTML elements with id/name attributes. Even when XSS payloads are sanitized (removing scripts and event handlers), an attacker can inject HTML elements that override JavaScript globals. This is especially dangerous when the overridden variable is later used in a security-sensitive context like a script src.

Bug bounty tip: Look for code that checks window.something || default and allows user-controlled HTML insertion (even sanitized). DOMPurify by default allows id and name attributes, which enables DOM clobbering.
