DOM XSS #1: The innerHTML Trap Easy

Objective

Execute alert('XSS') on this page by injecting code through the URL hash fragment.

The search page reflects your query from the URL hash into the DOM using innerHTML.

Hints

The page reads location.hash (everything after the # in the URL) and uses it as the search query. Try adding #test to the URL.
The search results are rendered using element.innerHTML = ... which parses HTML. Unlike textContent, innerHTML will execute embedded HTML tags.
Try: #<img src=x onerror=alert('XSS')> — The img tag fails to load, triggering the onerror handler which runs your JavaScript.

Vulnerable Code

// Source: URL hash fragment
var query = decodeURIComponent(
  location.hash.substring(1)
);

// Sink: innerHTML (dangerous!)
document.getElementById('results')
  .innerHTML =
    'Results for: ' + query + '';

How this works: The page reads the hash from the URL (everything after #) and displays it as search results. The hash is used directly in innerHTML without sanitization.

Real-world context: Many SPAs use hash-based routing and reflect parameters into the DOM. If innerHTML is used instead of textContent, it creates DOM XSS. This is one of the most common client-side vulnerabilities.
