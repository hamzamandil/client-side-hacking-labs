CSPT #1: Basic Path Injection Easy

Objective

The notifications page fetches /api/notifications/{userId} where userId comes from the URL. Use path traversal to make it fetch a different API endpoint and display sensitive data.

Goal: Make the page fetch /api/user/profile/admin instead of notifications, revealing the admin's email.

Hints

The page reads ?userId= from the URL and fetches /api/notifications/ + userId. If userId is 123, it fetches /api/notifications/123. What if userId contains ../?
Try ?userId=../user/profile/admin. The browser resolves /api/notifications/../user/profile/admin to /api/user/profile/admin. The ../ goes up one directory level.
The admin profile endpoint returns the admin's email, role, and other sensitive info. Use: ?userId=../user/profile/admin. The response data gets displayed in the notification area, leaking admin info.

Vulnerable Code

var userId = new URLSearchParams(
  location.search
).get('userId') || '1';

// VULNERABLE: userId injected into fetch path
fetch('/api/notifications/' + userId)
  .then(r => r.json())
  .then(data => {
    // Renders whatever the API returns
    renderData(data);
  });

Real-world context: Client-Side Path Traversal occurs when user input is concatenated into a fetch/XHR URL path without sanitization. Unlike server-side path traversal, the browser resolves ../ sequences BEFORE sending the request. So /api/notifications/../admin/users becomes /api/admin/users at the HTTP level.

Bug bounty tip: Search JavaScript for patterns like fetch('/api/' + param) or $.get('/endpoint/' + id) where the parameter comes from the URL, hash, or user input. Even if the server validates paths, the browser resolves traversals client-side.
