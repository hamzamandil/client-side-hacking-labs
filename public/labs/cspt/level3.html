CSPT #3: Response Injection via Upload Hard

Objective

Execute alert('XSS') by chaining CSPT with file upload. Upload a malicious JSON file, then use CSPT to make the app fetch your uploaded file instead of the real API. The app renders the response's title field via innerHTML.

Hints

1. The upload endpoint at /api/upload stores JSON and returns a file URL like /api/files/abc123
2. The product page fetches /api/settings/{productId} and renders the title field in innerHTML
Chain them: upload malicious JSON, then traverse to fetch your uploaded file.
Upload JSON with an XSS payload in the title field: {"section":"x","data":{"theme":"dark","language":"en","notifications":true},"title":"<img src=x onerror=alert('XSS')>"}. Note the file ID returned.
If your file is at /api/files/abc123, use CSPT to make the product page fetch it: ?productId=../files/abc123. The page fetches /api/settings/../files/abc123 which resolves to /api/files/abc123 — your malicious JSON. The title field renders via innerHTML = XSS!

Vulnerable Code

// Fetch product data
fetch('/api/settings/' + productId)
  .then(r => r.json())
  .then(data => {
    // Renders title from response (trusted)
    titleEl.innerHTML = data.title || data.section;
  });

// Upload accepts arbitrary JSON
POST /api/upload { content: "...", contentType: "..." }
// Returns: { fileId: "abc", url: "/api/files/abc" }

Real-world context: This is the Reverb technique from real bug bounty reports ($5K+ bounties). The attack chain:

  1. Find an upload endpoint that stores content you control
  2. Find a page that fetches an API endpoint using a user-controlled path
  3. Upload malicious JSON that matches the expected response structure
  4. Use CSPT to redirect the fetch to your uploaded file
  5. The app renders the poisoned response, triggering XSS
