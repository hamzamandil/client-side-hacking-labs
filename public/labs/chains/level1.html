Chain #1: Self-XSS + CSRF = Stored XSS Hard

Objective

The victim's bio field is vulnerable to XSS, but it's "self-only" — you can only edit your own bio. However, the profile update endpoint has no CSRF protection. Chain Self-XSS + CSRF to inject stored XSS into the victim's profile.

Goal: Make the victim's bio execute alert('XSS') when their profile is viewed.

Hints

The bio field renders with innerHTML. If you set your own bio to <img src=x onerror=alert('XSS')>, it executes when your profile is viewed. But this is "self-XSS" — only affects you. Not reportable alone.
The POST /api/user/profile/update endpoint accepts JSON, uses cookie-based auth, and has NO CSRF token. Any page can make this request on behalf of a logged-in user using fetch() with credentials:'include'.
Create an attacker page that, when visited by a logged-in victim, sends a CSRF request to update their bio with an XSS payload. Steps:
1. Log in as victim
2. Open the CSRF attack page
3. The attack page auto-submits a request to update the victim's bio
4. View the victim's profile — XSS fires!

Vulnerable Code

// Profile update: NO CSRF token check
app.post('/api/user/profile/update',
  requireAuth, (req, res) => {
    user.bio = req.body.bio; // Stored!
  }
);

// Profile display: innerHTML (XSS sink)
bioEl.innerHTML = userData.bio;

Real-world context: Self-XSS is often marked as "informational" or "won't fix" by programs. But when combined with CSRF, it becomes stored XSS affecting other users — a valid HIGH severity finding. The chain: any logged-in user who visits the attacker's page gets their profile infected with persistent XSS, which then fires for anyone viewing that profile.

Bug bounty tip: Whenever you find self-XSS, immediately check if the endpoint that stores the data has CSRF protection. If not, you have a valid stored XSS chain that most programs will accept.
