Chain #7: Open Redirect + OAuth + CSRF = ATO Expert

Objective

Chain THREE distinct vulnerabilities into a full account takeover:
1. Open redirect at /redirect?url=
2. OAuth redirect_uri manipulation
3. CSRF on email change endpoint
Steal the victim's OAuth code via redirect chain, then CSRF their email change.

Hints

Bug 1: /redirect?url= redirects to any URL (open redirect)
Bug 2: OAuth redirect_uri validation is prefix-based and allows path traversal
Bug 3: /api/user/change-email has no CSRF token
Each alone is low-medium severity.
1. Craft OAuth URL with redirect_uri that traverses to the open redirect
2. The open redirect sends the auth code to your attacker page
3. The OAuth flow also sets a session cookie
4. Use the session cookie to CSRF the email change
Triple chain = full ATO
The OAuth flow sets a session cookie when the victim authenticates. After redirect, the victim has a valid session. You can CSRF the email change using credentials: 'include'. So: OAuth popup → victim authenticates → session set → CSRF email change → ATO. The open redirect is the "glue" that makes the redirect_uri work.
