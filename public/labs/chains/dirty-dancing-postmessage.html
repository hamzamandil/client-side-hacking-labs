Hint: OAuth Dirty Dancing + PostMessage Leak

      
    

    

      
Three small decisions combine into a token theft: implicit flow, a callback page with a friendly analytics handler, and window.opener access. The attacker "dances" between the flow and the callback to pick up the token.


      

        
1
Find the bugs›

        

          Bug A — the OAuth server accepts response_type=token (implicit). Token lands in the URL fragment.

          Bug B — the callback page (/flowpay/sso-complete) installed a friendly analytics handler that responds to getAnalytics postMessage with location.href. Any caller can query it.

          Bug C — the callback also proactively sends location.href to window.opener via postMessage("*") so the opener always gets the URL.
        

      


      

        
2
Why the token ends up readable›

        

          With implicit flow, the browser redirects to /flowpay/sso-complete#access_token=eyJ.... The fragment stays in location.href. When the callback posts that URL back to the opener, everything after # comes with it.
        

      


      

        
3
Build the malicious URL›

        

          
/oauth/authorize?client_id=vuln-app&
response_type=token&
redirect_uri=/flowpay/sso-complete&
state=dancing&scope=read+write

          Open this in a popup via window.open. The victim is logged in → OAuth auto-approves → popup navigates to sso-complete with the token in the fragment.
        

      


      

        
4
Collect the token›

        

          Two ways:
          
    
            
  • Listen for window.addEventListener('message', ...) — the callback proactively sends {type:'flowpay:sso-done', url: ...} to opener.
    • 
            
  • Actively query: popup.postMessage({type:'getAnalytics'}, '*') — gadget replies with {type:'flowpay:analytics', url: ...}.
    • 
          

          Parse access_token= from the URL fragment. Done.
        

      


      

        
        

          
Open the attacker helper and click "Fire dirty-dancing chain". The page:

          
    
            
  • Opens the malicious URL in a popup
    • 
            
  • Polls it with postMessage({type:'getAnalytics'})
    • 
            
  • Receives the response with location.href including #access_token=...
