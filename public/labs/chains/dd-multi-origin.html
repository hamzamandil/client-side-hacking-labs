What the main app does

The Dash product team wanted an "analytics sandbox" to isolate third-party telemetry code. The analytics iframe is loaded from localhost:5000. To give the sandbox context (referrer, URL), the callback page sets iframe.name = JSON.stringify({href, referrer}) before loading. Reasonable, right?

Hint: Multi-Origin Token Leak

      
    

    

      
Three origins, three bugs. The glue that breaks everything is iframe.name.


      

        
1
Map the architecture›

        

          Main app on :3000 → opens OAuth on :4000 → redirects back to :3000 with token in fragment → :3000 loads analytics iframe from :5000. The iframe's origin has an XSS sink. Your JS ends up running there.
        

      

      

        
2
Find the leak primitive›

        

          Before setting iframe.src, the main app does iframe.name = JSON.stringify({href, referrer}). Inside the iframe, window.name is that string. Your XSS in the iframe reads window.name — which contains the token.
        

      

      

        
3
Find the XSS in the sandbox›

        

          /embed?src=XYZ reflects XYZ raw into a <script> tag. Anything you put there runs. The main app URL-encodes the telemetry query param and passes it through — so if you control telemetry, you get XSS on :5000.
        

      

      

        
4
Put it together›

        

          Malicious URL: /dd/callback?state=bad&telemetry=<your JS> reached via IdP implicit redirect. Your JS exfils window.name to the collector.
        

      

      

        
        

          Make sure you've signed into the IdP once (step 1). Then click "Fire exploit". The popup opens, OAuth auto-approves, the callback loads the sandbox with your JS, and the collector fills up within a second.
